状态与数据流建模是一种把系统理解为“状态变化 + 数据流动”的设计方法，用来描述对象如何变化、信息如何传播以及边界如何协作。

## Definition

当前来源明确提出：一个系统的本质可以被理解为“状态 + 数据流”。

这意味着设计系统时，至少要回答两类问题：

- 状态问题：系统里有哪些对象、它们处于哪些状态、会因哪些事件发生转移？
- 数据流问题：数据从哪里来、经过哪些处理、流向哪里、在哪些边界上被存储或转发？

## Building Blocks

| 元素 | 说明 | 例子 |
|---|---|---|
| 实体 | 系统中的核心对象 | 用户、会话、消息 |
| 状态 | 对象在某一时刻的可区分条件 | 在线 / 离线、已发送 / 已读 |
| 事件 | 触发状态变化的动作 | 发送消息、收到回执、心跳超时 |
| 转移 | 状态如何随事件变化 | 未送达 -> 已送达 -> 已读 |
| 数据流 | 数据在组件间的流动路径 | 客户端 -> 服务端 -> 存储 -> 推送 |
| 边界 | 系统划分出的职责界线 | 客户端、消息服务、状态服务、数据库 |

## Example from the Source

在聊天系统示例里：

- 用户模型关注 `userId`、昵称、在线状态和最后活跃时间。
- 会话模型关注会话类型、成员列表和创建时间。
- 消息模型关注消息内容、发送者、时间戳和状态。
- 系统流程被拆成“发送 -> 存储 -> 推送 -> 回执 -> 状态更新”的链路。

这类表达方式比“做一个聊天功能”更适合继续推导接口、存储模型和异常处理。

## Why It Matters

- 状态模型帮助团队识别乱序、重复写入、离线补偿和并发覆盖等问题。
- 数据流模型帮助团队看见系统依赖、延迟路径和持久化位置。
- 两者结合后，设计讨论会从“怎么写接口”升级为“系统如何运作”。

## Documentation Implications

- 系统设计文档适合至少给出状态表、关键事件和主要数据流路径。
- 当系统涉及可靠性要求时，应把消息状态、确认机制和失败补偿写成显式模型。
- 对复杂流程，流程图或时序图比纯文字描述更容易暴露遗漏边界。

## Common Misconceptions

- 只有分布式系统才需要状态机思维。
  只要对象会变化、行为会受上下文影响，就已经存在状态建模问题。
- 数据流只等于网络流量。
  数据流也包括进程内处理、队列传递、持久化和回放同步。
