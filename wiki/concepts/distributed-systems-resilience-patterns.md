分布式系统韧性模式是把理论上的“不确定性不可消除”转化为工程上的“系统仍可运行”的一组基础机制，核心包括超时、重试、幂等、Quorum、缓存一致性、限流、熔断、降级和故障转移。

## Definition

当前来源的工程层和决策层共同说明：分布式系统的可靠性不是靠“假设网络稳定”得到的，而是靠一系列明确的保护机制把失败限制在可接受范围内。

## Core Request-Safety Mechanisms

| 机制 | 解决什么问题 | 常见做法 |
|---|---|---|
| 超时 | 无法区分“慢”和“死” | 固定超时、动态超时、按 P99 调整 |
| 重试 | 瞬时失败导致请求未完成 | 指数退避、抖动、上限控制 |
| 幂等 | 请求重放造成重复副作用 | 请求 ID、去重表、状态机、版本检查 |

来源强调，这三者必须组合看：没有幂等的重试，会把临时故障变成数据错误。

## Read / Write Path Patterns

| 模式 | 主要收益 | 主要代价 |
|---|---|---|
| 同步写 | 更强的数据语义 | 延迟更高、故障时更脆弱 |
| 异步写 | 更低延迟、更高吞吐 | 可能丢最近写入 |
| 多数派确认 | 安全性和可用性的平衡 | 需要理解 Quorum 和副本交集 |
| 读主 / 读多数派 | 更强读一致性 | 热点集中、成本更高 |
| 读从 / 本地缓存 | 更低延迟、更高读吞吐 | 可能读到旧数据 |
| Cache-Aside | 实现简单、适合高读场景 | 需要处理缓存失效与并发窗口 |

## Availability Protection

| 机制 | 作用 | 当前来源中的典型表达 |
|---|---|---|
| 限流 | 防止系统被流量压垮 | 固定窗口、滑动窗口、令牌桶、漏桶 |
| 熔断 | 防止故障扩散 | Closed / Open / Half-Open 状态机 |
| 降级 | 保住核心路径 | 返回默认值、缓存值、关闭非核心功能 |
| 故障转移 | 组件失效时恢复服务 | 心跳检测、选主、路由切换、客户端重连 |

## Transaction and Coordination Heuristic

来源给出的建议很明确：

- 能用单库事务解决，就先不要上分布式事务。
- 能接受最终一致，就优先用补偿或异步流程。
- 只有在必须强一致时，才考虑 2PC/3PC 这类高代价方案。

这是一种“先降低协调面，再增加协议复杂度”的设计原则。

## Design Review Checklist

当前来源适合沉淀为如下评审问题：

- 数据会不会丢？在哪些故障窗口会丢？
- 请求会不会重复执行？幂等键和去重边界在哪里？
- 失败后是重试、回滚、补偿还是降级？
- 读路径是否允许旧数据？缓存如何失效？
- 有没有单点？如何做自动恢复和人工回滚？
- 容量或流量增长 10 倍时，瓶颈会出现在什么位置？

## Why It Matters

- 它把“高可用”从一句目标口号，拆成一组可以落地实现和审计的机制。
- 它帮助团队在异常路径上先设计、先压测、先观测，而不是上线后再补救。
- 它与一致性模型、共识和复制策略是同一条链上的不同层：上层给语义，下层给实现，中间靠这些机制承受现实世界的失败。

## Documentation Implications

- 接口文档应写明超时、重试、幂等键和错误分类，而不只是列出 HTTP 状态码。
- 存储文档应写出读写路径、确认级别、缓存策略和故障转移触发条件。
- Runbook 应覆盖限流、熔断、降级和主备切换，而不是只在报警后手工排查。
- 设计文档适合把“正常路径”和“失败路径”并列呈现。

## Common Misconceptions

| 误解 | 更准确的理解 |
|---|---|
| 重试总能提高成功率 | 没有退避和幂等时，重试可能放大故障 |
| 缓存只是性能优化 | 缓存会直接改变一致性窗口和失效路径 |
| 熔断 / 限流 / 降级是上线后再补的运维细节 | 它们是架构设计的一部分 |
| 有主备切换就等于高可用 | 没有检测、路由更新、客户端重连和数据语义，高可用只是表面拓扑 |
