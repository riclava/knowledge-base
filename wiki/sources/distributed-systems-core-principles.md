## Core Thesis

- 分布式系统的起点不是“如何把单机做大”，而是先接受网络、时间和节点状态都带来根本不确定性。
- CAP、FLP、PACELC、拜占庭容错等理论不是考试知识点，而是告诉工程师哪些目标不能同时保证、哪些代价必须显式承担。
- 一致性模型、共识算法、复制/分片和逻辑时钟是理解大多数分布式系统设计的核心抽象层。
- 真正把系统做成产品，靠的是超时、重试、幂等、Quorum、缓存、限流、熔断、降级和故障转移这些工程机制。
- 架构设计的本质是基于业务优先级做取舍，而不是寻找一个“既强一致、又高可用、又低延迟、又低复杂度”的完美方案。

## Key Theoretical Threads

### 1. 不确定性不是边缘情况，而是默认前提

来源把“无法区分对方是慢还是死”作为第一性原理，并用两将军问题说明确认链条永远无法做到绝对确定。

### 2. 理论边界直接决定工程行为

| 理论 | 来源中的定位 | 直接工程含义 |
|---|---|---|
| CAP | 网络分区时必须在一致性和可用性之间选择 | 故障时必须定义“拒绝服务”还是“允许旧数据” |
| FLP | 异步系统中无法保证确定性共识一定终止 | 需要超时、重试、部分同步假设或随机化 |
| PACELC | 无分区时仍需在延迟和一致性之间权衡 | 正常路径设计也必须做 trade-off，而不是只盯故障态 |
| 拜占庭容错 | 恶意节点场景比崩溃故障难得多 | 内部系统通常采用 crash-fault 模型，区块链等再考虑 BFT |

### 3. 抽象模型帮助工程师从产品实现回到原理

- 一致性模型回答“读能看到什么”。
- 共识算法回答“多个节点如何对一个值或日志顺序达成一致”。
- 复制和分片回答“数据如何放置、如何扩展、如何迁移”。
- 逻辑时钟回答“没有可信全局时钟时，如何表达先后关系和并发”。

## Engineering Patterns Highlighted

| 机制 | 来源中的作用 | 关键提醒 |
|---|---|---|
| 超时 | 把“慢和死无法区分”转成可操作决策 | 超时过短会误判，过长会拖累恢复 |
| 重试 | 提高瞬时故障下的成功率 | 必须配合指数退避和抖动，避免重试风暴 |
| 幂等 | 防止请求重放造成重复副作用 | 唯一请求 ID、去重表、状态机都很重要 |
| Quorum 读写 | 用副本交集换取一致性保障 | `W + R > N` 是核心关系，不等于所有场景都该多数派读写 |
| Cache-Aside | 在性能和一致性之间取得工程平衡 | 更新数据库后删缓存比直接改缓存更稳妥 |
| 限流 / 熔断 / 降级 | 在压力或故障中保住核心路径 | 这是系统韧性设计，不是“可选优化” |
| 故障转移 | 在主节点故障时恢复服务 | 检测、选主、路由更新和客户端重连必须连成闭环 |

## Design Heuristics

来源给出了一套比较完整的架构决策顺序：

1. 先明确业务优先级，是数据正确性、高可用、低延迟还是高吞吐优先。
2. 再选择一致性模型，而不是反过来让技术方案决定业务规则。
3. 再设计失败处理，包括哪些错误可重试、哪些操作必须幂等、部分成功如何补偿。
4. 然后确定数据分布、读写路径、缓存策略和扩容方式。
5. 最后才评估是否真的需要分布式事务，并尽量优先用单库事务或最终一致替代。

## Documentation Implications

- 分布式系统文档应把“失败时系统怎么表现”写成一等信息，而不是只写正常流程。
- 一致性承诺应写成面向业务的语言，例如“余额绝不返回旧值”或“评论可能延迟数秒可见”。
- 读写路径文档应显式写出确认级别、重试策略、幂等键、缓存失效方式和故障降级策略。
- 架构评审文档适合直接复用来源里的正确性 / 可用性 / 性能 / 可扩展性 / 可运维性清单。

## Caveats and Boundaries

- 这份来源更像教学总览，而不是某个具体系统的正式设计文档或协议规范。
- 其中部分产品归类是教学化简，例如把某些数据库直接归到单一 CAP 位置；正式资料里仍需回到各产品的具体配置和语义。
- 来源提到 2PC/3PC、Saga、CRDT、Kafka、Spanner、Dynamo 等，但主要用于定位概念，没有展开机制细节。
- 可观测性、测试验证和 Jepsen 式故障注入在当前来源中只被轻触，没有成为完整专题。
