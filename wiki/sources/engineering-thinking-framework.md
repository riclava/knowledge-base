## Core Thesis

- 技术研发的本质不是“写代码”，而是把现实问题转化为可计算系统。
- 工程师的关键价值之一，是在实现之前先把问题说清楚、模型建清楚、边界划清楚。
- 写代码只是最后一步，前面还需要经历 `现实问题 -> 抽象 -> 模型 -> 系统 -> 代码` 的递进过程。
- 验证与迭代不是收尾动作，而是降低设计风险的前置动作。

## Thinking Framework

| 能力 | 核心问题 | 产出 |
|---|---|---|
| 抽象能力 | 这个问题里真正稳定的结构是什么？ | 实体、关系、关键约束 |
| 建模能力 | 系统中的状态、事件和转移是什么？ | 状态机、数据结构、行为规则 |
| 系统拆解能力 | 数据、流程和边界如何分开？ | 模块划分、职责边界、接口草图 |
| 验证与迭代 | 在写代码前怎样确认设计站得住？ | 手推结果、流程图/时序图、极端 case 清单 |
| 工程决策能力 | 约束下该怎么做取舍？ | 方案比较、风险判断、扩展路径 |

## Knowledge-Graph View of Engineering Growth

来源把工程成长拆成四层认知：

1. 理解问题：用抽象思维和分治思维把现实问题变成可讨论对象。
2. 构建模型：把系统理解为状态变化和数据流动。
3. 设计系统：通过分层、接口和封装定义稳定边界。
4. 工程决策：在权衡与验证中控制风险，而不是盲目追求“最优解”。

其中“知识图谱”并不是特指某种产品能力，而是强调要把零散 CS 知识组织成可调用的思维网络。

## Worked Example: Chat System Design

来源用“类似微信的聊天系统”演示了完整思考链路：

- 先定义目标：稳定、低延迟、可追踪、可恢复，而不只是“能发消息”。
- 再抽象对象：用户、会话、消息、会话成员、消息回执。
- 接着建模：定义用户状态、消息状态、对象关系与在线机制。
- 然后做系统设计：客户端、接入层、消息服务、状态服务、数据库分层。
- 最后处理关键问题：可靠送达、并发顺序、离线补发、在线状态维护、扩展性预留。

这个例子说明，系统设计题不是考“记住哪个架构名词”，而是考能否把问题逐步转成可实现、可验证的结构。

## Documentation Implications

- 设计文档应显式写出问题定义、核心对象、状态与数据流、系统边界和验证方案，而不是只给接口列表。
- 培训材料适合采用“概念 + 模型 + 例题拆解”的结构，让抽象能力和建模能力可迁移。
- 当主题是 AI 协同时，更需要工程师把问题、上下文、边界和验收标准表达清楚，因为 AI 不会替代这些前置判断。
- 系统设计类文档适合附带极端 case、失败场景和验证清单，帮助团队在编码前统一理解。
