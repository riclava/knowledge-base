分布式系统里，一次业务往往横跨多个服务/数据库，"要么全成功要么全失败"变得极其困难。这一篇讲透 CAP/BASE 理论、常见分布式事务方案（2PC/TCC/Saga/消息表）、TCC 的三大落地难题、幂等设计、以及分布式 ID 生成。

### ⚖️ CAP 与 BASE

**CAP**：一致性(C)、可用性(A)、分区容错(P) 三者不可兼得。分布式系统网络分区（P）必然存在，所以实际是在**分区发生时**在 C 和 A 之间取舍（CP 还是 AP）。

**BASE**：Basically Available（基本可用）、Soft state（软状态/中间态）、Eventually consistent（最终一致）——是对强一致的工程妥协。

🧠 **类比**：CAP 像"异地两个仓库网断了"——要么坚持"两边数据必须一致，宁可暂停服务"（CP），要么"先各自照常营业，等网通了再对账"（AP）。BASE 就是"允许暂时对不上账，但最终一定对上"。

**关键澄清**：不是"永远三选二"，平时网络正常 C 和 A 都能满足，CAP 的取舍只在分区那一刻发生。

### 🧭 常见分布式事务方案

*   **2PC/3PC（XA）**：强一致，协调者统一提交/回滚，但同步阻塞、协调者单点、性能差。
*   **TCC（Try-Confirm-Cancel）**：业务层两阶段，Try 预留资源、Confirm 确认、Cancel 补偿；性能好但业务侵入强、需处理幂等/空回滚/悬挂。
*   **Saga**：长事务拆成一串本地事务，每步配一个补偿操作，出错反向补偿；最终一致，适合流程长、跨多服务。
*   **本地消息表 / 事务消息**：异步最终一致（见 [[消息队列-Kafka与RabbitMQ]]）。

🧠 **类比**：2PC 像"婚礼司仪喊'大家都准备好了吗？'全答好才继续，中间谁没准备好全体等着"（阻塞）；TCC 像"订酒店先'预留房间'（Try），确认了才'正式入住'（Confirm），取消就'释放房间'（Cancel）"；Saga 像"多步旅行，某步出问题就一步步往回退票退房（补偿）"。

**选型口诀**：能最终一致就别上强一致。强一致（2PC/TCC）代价高，多数互联网业务用 Saga/消息表做最终一致。

### 🧨 TCC 的空回滚、悬挂、幂等

*   **空回滚**：Try 还没执行（如网络问题没到），Cancel 先到了 → 记录事务状态，发现没 Try 过就直接返回成功，不做实际回滚。
*   **悬挂**：Cancel 已执行完，迟到的 Try 才到 → 检测到该事务已 Cancel，就拒绝执行 Try（否则预留的资源没人释放）。
*   **幂等**：Confirm/Cancel 可能重复调用 → 用全局事务 id + 状态表去重。

🧠 **类比**：空回滚像"还没下单就收到退款请求，系统得认出'这单根本没下过'，别真去退不存在的款"；悬挂像"退款都处理完了，下单请求才姗姗来迟，得拦住它别再占资源"。

这三个问题是 TCC 落地的核心难点，靠"事务状态表 + 全局事务 id"统一解决。

### 🔁 幂等设计的通用手段

数据库唯一约束、去重表/去重 key、状态机流转（只允许合法状态转移）、token 令牌（先领 token 再提交，用过即失效）、乐观锁版本号。

🧠 **类比**：幂等像"防止重复提交订单"——发一张一次性令牌（token），提交时核销，重复提交时令牌已失效就拦下。

幂等是分布式重试/消息去重的基石，几乎所有"可能重试"的接口都要设计幂等。

### 🆔 分布式 ID 生成方案

*   **UUID**：本地生成无依赖，但无序（做主键导致 B+ 树频繁页分裂、索引大）。
*   **雪花算法（Snowflake）**：时间戳 + 机器 id + 序列号，趋势递增、性能高，但依赖时钟。
*   **号段模式**：从 DB 批量取一段 id 缓存本地发放，减少 DB 压力。
*   **Redis incr**：简单但依赖 Redis。

**雪花时钟回拨问题**：机器时间被回拨会生成重复 id → 检测到回拨则等待/拒绝/借用备用位。

🧠 **类比**：UUID 像"随机长串车牌"，唯一但杂乱不好排序；雪花 id 像"按时间+窗口编号的排队号"，基本递增又不重复；号段模式像"售票窗口一次领 1000 张号，发完再领"，省得每张都去总台要。

做数据库主键优先选趋势递增的（雪花/号段），避免 UUID 导致的写入性能问题。

### 📌 记忆卡

*   **CAP**：分区时才在 C 和 A 间取舍；平时两者都能满足。
*   **BASE**：基本可用 + 软状态 + 最终一致。
*   **事务方案**：2PC 强一致阻塞、TCC 业务侵入、Saga 补偿、消息表异步；能最终一致就别强一致。
*   **TCC 三难**：空回滚、悬挂、幂等 → 事务状态表 + 全局事务 id。
*   **幂等手段**：唯一约束、去重表、状态机、token、乐观锁。
*   **分布式 ID**：主键用趋势递增（雪花/号段），别用 UUID。
