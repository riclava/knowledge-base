Go 语言底层与运行时，是理解一切 Go 高性能特性的根基。你可以把运行时（runtime）想象成一个**藏在你代码背后的操作系统**：它自己管理调度、内存分配、垃圾回收，让你写出来的简单代码能在多核机器上高效跑起来。这一篇把最常被问到的底层机制串起来讲透。

### 🧩 GMP 调度模型：P 到底是干什么的

GMP 是 Go 调度器的核心，三个角色各司其职：

*   **G (Goroutine)**：任务本身，Go 里并发执行的最小单位，每个只占几 KB 栈，可以轻松开成千上万个。
*   **M (Machine)**：真正干活的操作系统线程（OS Thread），是"工人"。
*   **P (Processor)**：调度上下文，相当于"工位/许可证"。数量默认等于 `GOMAXPROCS`（通常 = CPU 核数）。

关键规则：**M 必须先拿到一个 P 才能执行 G**。每个 P 有自己的本地可运行队列（runq，容量 256），调度取任务的优先级是：本地队列 → 全局队列 → netpoll → work-stealing（空闲 P 去别的 P 偷一半 G）。

🧠 **类比**：把 P 想成"工位"，M 是"工人"，G 是"待办任务"。工位数量固定（GOMAXPROCS），工人再多也只有拿到工位才能干活；每个工位有自己的任务篮（本地队列），篮子空了就去别人篮子里"偷"一半任务，避免有人闲死有人累死。

**为什么要引入 P？** 早期只有 GM 模型，所有 G 抢一把全局锁，锁竞争严重、无本地缓存。加了 P 就有了"每工位一个任务篮"，大幅降低锁争用。这是 GMP 相较早期 GM 的核心进化。

### ⚙️ goroutine 阻塞时调度器怎么处理

不同的阻塞类型，调度器的应对策略完全不同，这是高频追问点：

*   **阻塞式系统调用**（如读文件）：M 会被系统调用卡住，于是 M 和 P 解绑（handoff），P 交给另一个空闲/新建 M 继续跑其他 G，避免整个工位闲置。**这种阻塞会占用 M**，所以可能触发新建线程。
*   **网络 IO**：交给 netpoller（底层 epoll/kqueue）托管，G 被挂起但**不占用 M**，数据就绪后 G 重新入队。这是 Go 高并发网络的关键。
*   **channel 阻塞**：G 进入 channel 的等待队列（sudog），M 立刻去跑别的 G，线程不阻塞。

🧠 **类比**：工人（M）去办一件"必须现场排队"的事（阻塞系统调用），就把工位（P）让给另一个工人接着干别的活；而网络 IO 像"取号后回来等叫号"，人不用一直站队里（不占 M），号到了再回来。

记忆要点：**阻塞系统调用占 M，网络 IO 走 netpoller 不占 M**。

### 🚦 抢占式调度与 sysmon

早期 Go 是**协作式抢占**——只在函数调用的栈检查点让出，一个没有函数调用的死循环（`for {}`）会饿死其他 G，甚至卡死 GC（STW 等不到抢占点，这是 1.14 前的经典 bug）。Go 1.14 起改为**基于信号的异步抢占**：运行时给线程发信号，强制正在跑的 G 让出。

**sysmon** 是后台监控线程，它不绑 P、独立运行，职责包括：抢占运行超过 10ms 的 G、回收陷入系统调用过久的 P、定期触发 GC、执行 netpoll。

🧠 **类比**：sysmon 像"车间巡检员"，看到某个工人霸占工位太久（>10ms），就拍他肩膀强制换人，保证任务轮转公平。

### 📦 slice 与 array：引用视图 vs 值类型

*   **array**：值类型、长度固定，赋值/传参是整份拷贝。
*   **slice**：引用视图，底层是结构体 `{ptr 指向底层数组, len 长度, cap 容量}`。

**扩容规则**（简化）：期望容量 > 2×旧 cap 直接用期望值；旧 cap < 256 翻倍；≥256 每次约 1.25 倍平滑增长。扩容会分配**新数组并拷贝**，原 slice 与新 slice 从此脱钩。

🧠 **类比**：array 像"固定格子的冰格"，大小定死；slice 像"指向某段冰格的便利贴"，写着起点、已用几格、总共几格。扩容就像格子不够了换个更大的冰盒并把冰倒过去。

**共享底层数组的坑**：`b := a[1:3]` 后对 b 的 append 若未超 cap 会**改到 a 的数据**；截取大数组的小 slice 会导致大数组无法 GC（内存泄漏）。

### 🗺️ map 底层：桶 + 渐进式扩容

*   **结构**：hmap + 若干 bucket（bmap），每个 bucket 存 8 个 key/value，放不下用溢出桶（overflow）链接；负载因子超阈值触发**渐进式扩容**（增量搬迁 oldbuckets，不一次性搬完，避免卡顿）。
*   **无序**：遍历时随机选起始 bucket 和槽位，故意防止你依赖顺序。
*   **并发读写**会触发 `fatal error: concurrent map read and map write`（连 recover 都拦不住），必须用 `sync.Map` 或加锁。

🧠 **类比**：map 像"一排储物柜（bucket），每个柜子 8 个格子"。key 经哈希决定进哪个柜子，柜子满了就挂个"加长抽屉"（溢出桶）。搬家（扩容）不是一次搬完，而是每次访问顺手搬一点（渐进式）。

`sync.Map` 适合"读多写少 / key 相对固定"，内部用 read + dirty 双 map 减少锁；写多场景不如"分片锁 map"。

### 🎁 interface 底层：iface 与 eface，以及 nil 陷阱

*   **eface**（空接口 `interface{}`/`any`）：`{_type 类型指针, data 数据指针}`。
*   **iface**（带方法的接口）：`{itab, data}`，itab 里含具体类型信息 + 方法地址表。

**nil 陷阱**：接口只有**类型和值都为 nil** 时才 `== nil`。若把一个 `(*T)(nil)` 赋给接口，接口带了类型信息，`i == nil` 就是 false。

🧠 **类比**：接口像"带标签的盒子"——标签写"里面装的是什么类型"，盒子里放"数据地址"。哪怕盒子里是空的（data=nil），只要贴了标签（有类型），它就不等于"完全的空盒子（nil）"。

**经典坑**：函数返回 `error` 接口，内部 `var err *MyErr = nil; return err`，调用方 `if err != nil` 恒成立。修复：返回时判断或直接 `return nil`。

### 🥞 defer 的执行顺序与性能

*   多个 defer 按 **LIFO（后进先出）** 执行。
*   defer 的**参数在注册那一刻就求值**（不是执行时）。
*   1.13 引入 defer 池化、1.14 引入 open-coded defer（编译期内联展开），开销已接近普通函数调用。

🧠 **类比**：defer 像"叠盘子"——最后叠上去的盘子最先被拿走（LIFO）。参数求值像"贴便签时就把当时的值写死了"，后面变量再变也不影响便签内容。

**坑**：循环里 `defer` 会累积到函数结束才释放（如循环内 `defer file.Close()` 导致文件句柄堆积），应在循环体内显式关闭或封装成独立函数。

### 🎈 逃逸分析：变量到底在堆还是栈

编译期分析变量生命周期是否"逃出"函数栈帧。栈分配随函数返回自动回收（快、无 GC 负担）；逃逸到堆则交给 GC 管理。

*   **查看**：`go build -gcflags="-m"`（会打印 escapes to heap）。
*   **常见逃逸原因**：返回局部变量指针、赋值给 interface（装箱）、闭包捕获引用、变量太大放不下栈、编译期大小不确定（如变长 slice）。

🧠 **类比**：栈像"随手用完就丢的草稿纸"（函数返回即撕掉），堆像"要归档、由清洁工（GC）定期清理的档案柜"。变量如果被外面还要用（返回指针/被闭包抓住），就只能进档案柜。

**减少逃逸 = 减少堆分配 = 降低 GC 频率**，是性能优化核心手段之一；但不要为此写反直觉代码，先测量再优化。

### 📮 值接收者 vs 指针接收者

*   **值接收者** `func (t T)`：`T` 和 `*T` 都能调用，方法内操作的是**副本**。
*   **指针接收者** `func (t *T)`：能修改原对象；`*T` 的方法集包含指针方法，而 `T`（值）的方法集**不含**指针方法——直接影响接口实现判定。

🧠 **类比**：值接收者像"给你一张复印件"，怎么改都不影响原件；指针接收者像"给你原件的地址"，能改原件。想改数据 / 结构体大 → 用指针。

**规范**：一个类型的方法要么统一用值、要么统一用指针（避免混用引发方法集混乱）；**含 Mutex 的结构体必须用指针接收者**（值拷贝会复制锁，导致锁失效）。

### 🎯 context：取消与超时的传播

*   **作用**：跨 API 和 goroutine 传递取消信号、超时/截止时间、请求作用域的值（trace id 等）。
*   **用法**：`WithCancel/WithTimeout/WithDeadline/WithValue`；返回的 cancel **必须调用**（`defer cancel()`）否则泄漏；约定作为函数**第一个参数**。

🧠 **类比**：context 像"包工头手里的哨子 + 闹钟"——一吹哨（cancel）所有干活的工人（goroutine）都停手；闹钟响（timeout）自动吹哨。哨声会一层层传给所有下级（父 ctx 取消，子 ctx 全取消）。

`WithValue` 只放请求域元数据，别用来传函数参数；key 用自定义类型避免碰撞。

### 🧰 补充：几个易混点

**new vs make**：`new(T)` 分配零值内存返回 `*T`，适用于任意类型；`make` 只用于 slice/map/channel，返回初始化好的**引用类型本身**（非指针）。类比：`new` 是"买个空盒子给你钥匙（指针）"，`make` 是"买来并组装好能直接用的家电"。

**string vs []byte**：string 不可变（底层 `{ptr, len}`），`[]byte` 可变。互转默认会**拷贝内存**（保证不可变性）。高频转换用 `strings.Builder`、复用 buffer 或谨慎用 `unsafe` 避免拷贝。类比：string 是"刻在石碑上的字"，[]byte 是"白板上的字"，互转就是抄一份。

**init 与包初始化顺序**：依赖包先初始化 → 包级变量按依赖顺序 → `init()`（一个文件可多个，按出现顺序；多文件按文件名字典序），全部在 main 前执行。类比：像装修房子——先通水电（依赖包）、再摆家具（变量）、最后收尾检查（init），才能入住（main）。

### 📌 记忆卡

*   **GMP**：G 任务、M 工人、P 工位；P 数量 = GOMAXPROCS；work-stealing 偷一半。
*   **阻塞**：系统调用占 M（可能新建线程），网络 IO 走 netpoller 不占 M。
*   **抢占**：1.14 起基于信号的异步抢占；sysmon 巡检 >10ms 的 G。
*   **slice**：`{ptr,len,cap}`；扩容拷贝新数组；共享底层数组有坑。
*   **map**：桶存 8 个 kv + 溢出桶 + 渐进式扩容；无序；并发读写直接 fatal。
*   **interface**：eface/iface；类型和值都 nil 才 == nil。
*   **逃逸**：被外部还要用就逃到堆；减少逃逸减轻 GC。
*   **接收者**：想改/大结构体/含锁 → 指针。
