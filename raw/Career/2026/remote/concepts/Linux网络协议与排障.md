后端专家不仅要懂代码，还要懂代码跑在什么之上——TCP/IP、Linux 系统、以及出问题时的排障套路。这一篇讲透 TCP 握手挥手、CLOSE_WAIT/TIME_WAIT、可靠传输与拥塞控制、HTTP 各版本、常用排障命令、CPU/Load 定位、fd 耗尽、零拷贝与 epoll。

### 🤝 TCP 三次握手、四次挥手

*   **三次握手**（SYN → SYN+ACK → ACK）：确认双方收发能力都正常，防止历史失效的连接请求突然到达而错误建连。
*   **四次挥手**：TCP 全双工，两个方向要各自关闭，被动方收到 FIN 后可能还有数据要发，所以 ACK 和 FIN 分开发，共四次。

🧠 **类比**：三次握手像打电话——"喂能听到吗？（SYN）""能，你能听到我吗？（SYN+ACK）""能，开始说吧（ACK）"，双方都确认了收发正常。四次挥手像"我说完了（FIN）→ 我知道了但我还没说完（ACK）→ ...我也说完了（FIN）→ 好的挂了（ACK）"。

**为什么不是两次握手**——防止旧的重复连接请求让服务端错误建连、浪费资源。TIME_WAIT（2MSL）保证最后的 ACK 可靠到达 + 让旧报文消散。

### 🚨 大量 CLOSE_WAIT 说明什么

CLOSE_WAIT 是"对方已发 FIN 要关，但我方还没调用 close()"的状态。大量堆积 = **应用代码没正确关闭连接**（忘了 Close、连接被阻塞无法关、连接泄漏）。

🧠 **类比**：对方说"我要挂电话了（FIN）"，你"嗯"了一声（ACK）但一直不挂自己这头（不 close），听筒就一直占着（CLOSE_WAIT 堆积），线路（fd）迟早被占满。

排查方向是代码——response body 没 close、DB/HTTP 连接没归还、defer close 被跳过。这是"我方 bug"的强信号。（对比：**TIME_WAIT 多**是主动关闭方的正常现象。）

### 📡 TCP 可靠传输与拥塞控制

*   **可靠**：序列号 + 确认应答（ACK）、超时重传、滑动窗口、流量控制（接收方通告窗口大小防被压垮）。
*   **拥塞控制**：慢启动（指数增长探测）、拥塞避免（线性增长）、快重传、快恢复。

🧠 **类比**：拥塞控制像"新手上高速"——先慢慢加速试探路况（慢启动），感觉顺就稳步加速（拥塞避免），一旦发现前方拥堵（丢包）马上减速（快恢复），避免堵死整条路。

**辨析**：流量控制是"别把接收方缓冲区撑爆"（点对点），拥塞控制是"别把网络整体压垮"（全局）。

### 🌐 HTTP/1.1、HTTP/2、HTTP/3

*   **1.1**：文本协议、长连接（keep-alive）但有队头阻塞（一个响应慢堵住后面）。
*   **2**：二进制分帧、多路复用（一个连接并发多请求）、头部压缩（HPACK）、服务端推送。
*   **3**：基于 QUIC（UDP），解决 TCP 层的队头阻塞、连接迁移快、握手更快。

🧠 **类比**：1.1 像"单车道，前车慢后车全堵"；2 像"多车道但共用一条路基，路基（TCP）出坑所有车道都受影响（TCP 队头阻塞）"；3 换成"每条车道独立路基（UDP+QUIC）"，一条坑了不影响其他。

HTTP/2 解决了应用层队头阻塞，但 TCP 层丢包仍会阻塞所有流，HTTP/3 用 QUIC 彻底解决。

### 🧰 常用排障命令

`top/htop`（CPU/内存/进程）、`vmstat`（系统整体）、`iostat`（磁盘 IO）、`free`（内存）、`ss/netstat`（连接状态）、`tcpdump`（抓包）、`lsof`（打开的文件/连接）、`strace`（系统调用跟踪）、`perf`（性能剖析）、`dmesg`（内核日志，看 OOM kill）、`pidstat`。

🧠 **类比**：这些命令像"医生的听诊器、血压计、X 光机"——top 量整体状态，iostat 查磁盘"血压"，tcpdump 是"网络 X 光"，strace 是"看它到底在跟系统要什么"。

**记忆分类**：看 CPU（top/pidstat）、看内存（free/vmstat）、看 IO（iostat/iotop）、看网络（ss/tcpdump）、看行为（strace/lsof）。

### 🔥 CPU 打满如何逐步定位

`top` 找占用高的进程 → `top -Hp <pid>` 找占用高的线程 → 结合 pprof（Go）/ perf / 火焰图定位到函数。看 CPU 构成：`us`（用户态）高 → 看应用代码；`sy`（内核态）高 → 看系统调用/上下文切换；`wa`（IO 等待）高 → 看磁盘/网络 IO。

🧠 **类比**：像医生查"发烧原因"——先定位哪个器官（进程）发热，再细查哪块组织（线程/函数），再判断是自身炎症（us 用户态代码）还是外部感染（sy 系统调用/wa IO）。

Go 服务直接上 pprof cpu profile + 火焰图，比 perf 更精准到函数。

### ⚖️ Load 高但 CPU 不高

Load 统计的是"可运行 + 不可中断（D 状态，通常等 IO）"的进程数。CPU 不高但 load 高，多半是大量进程卡在 IO 等待（磁盘慢、网络慢、NFS 卡）。用 `iostat`、`iotop`、看 D 状态进程排查。

🧠 **类比**：load 高 CPU 不高像"餐厅排长队但厨师没在忙"——不是厨房（CPU）慢，是大家都在等某个上游供货（IO），队伍（load）越排越长。

Linux 的 load 包含 D 状态（不可中断睡眠）是它区别于其他系统的关键，所以 load 高 ≠ CPU 忙。

### 📁 文件描述符耗尽（too many open files）

调高 `ulimit -n` / systemd LimitNOFILE；用 `lsof -p <pid>` 查是什么占满（连接泄漏？文件没关？）；根治靠复用连接池 + 确保资源关闭。

🧠 **类比**：fd 像"公司的门禁卡总数"——发完了新人进不来（无法新建连接/打开文件）。要么多印卡（调 ulimit），要么查是谁拿了卡不还（连接泄漏）。

每个 socket/文件都占 fd，连接泄漏（见 CLOSE_WAIT）是常见根因。

### 📤 零拷贝、mmap、sendfile

传统读文件发网络要经历"磁盘 → 内核缓冲 → 用户缓冲 → socket 缓冲 → 网卡"多次拷贝和上下文切换。零拷贝（sendfile/mmap）省去用户态那趟，数据在内核内直接流转。

🧠 **类比**：普通方式像"仓库发货先搬到你办公室，你再搬去快递点"（多搬一趟）；零拷贝像"仓库直接发给快递点"，省掉中间搬运。

Kafka、Nginx 静态文件、文件下载服务都靠零拷贝提升吞吐。

### 🎛️ epoll vs select/poll

select/poll 每次调用都要把所有 fd 传入内核并**遍历全部**判断谁就绪（O(n)）；epoll 用事件回调 + 就绪列表，只返回真正就绪的 fd（O(1)），支持海量连接。

🧠 **类比**：select 像"点名时把全班名字从头念一遍看谁在"（人越多越慢）；epoll 像"谁到了自己在签到表登记，老师只看签到表"（不管班多大都高效）。

epoll 是 Go netpoller、Nginx、Redis 高并发网络的底层基础（Linux 上）。

### 📌 记忆卡

*   **三次握手**确认双向收发；**四次挥手**因全双工各关一次；TIME_WAIT=2MSL。
*   **CLOSE_WAIT 多** = 我方没 close（代码 bug）；**TIME_WAIT 多** = 主动关闭方正常。
*   **可靠传输**：序列号 + ACK + 重传 + 滑动窗口；拥塞控制 = 慢启动/避免/快重传/快恢复。
*   **HTTP**：1.1 队头阻塞、2 多路复用（TCP 层仍阻塞）、3 QUIC 彻底解决。
*   **CPU 定位**：top → top -Hp → pprof/火焰图；看 us/sy/wa。
*   **Load 高 CPU 不高** = 大量进程等 IO（D 状态）。
*   **fd 耗尽**：调 ulimit + 查连接泄漏。
*   **epoll** O(1) 只返回就绪 fd；零拷贝省用户态拷贝。
