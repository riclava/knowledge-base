线上排障能力最能体现一个后端的"实战功力"。这一篇讲透接口大面积超时、OOM、goroutine 暴涨、连接池打满这几个高频故障的排查思路，以及一份合格复盘（Postmortem）该包含什么。贯穿始终的一条铁律：**先止血，再定位**。

### 🚑 接口大面积超时的排查思路

**分层定位**：监控大盘看现象（RT/错误率/QPS 突变）→ 判断是自身还是下游（DB/缓存/RPC/MQ 哪层慢，看链路追踪）→ 查资源（CPU/内存/GC/连接池/FD/网络）→ 查变更（最近发布？配置改动？流量突增？）。

**原则**：先止血（回滚/降级/限流/扩容）恢复业务，再从容定位根因——别在故障时纠结根因让损失扩大。

🧠 **类比**：像医院急诊——先止血保命（恢复服务），稳定后再做详细检查找病因（根因分析）。慌乱中最忌讳"边流血边研究为什么流血"。

**排查口诀**：看现象 → 分自身/下游 → 查资源 → 追变更。**80% 的线上问题和"最近的变更"有关，先看变更效率最高**。

### 💥 服务频繁 OOM 被 kill

pprof heap 看内存分布；核对容器 memory limit 与 GOMEMLIMIT/GOGC 配比；查大对象、goroutine 泄漏、无上限的缓存/map、大量并发导致的瞬时分配；`dmesg` 看是否被 OOM killer 杀。

🧠 **类比**：OOM 像"房间被塞爆"——先看是"长期堆积（泄漏）"还是"瞬间涌入太多东西（并发高峰分配）"，再决定是修漏还是限流/扩容/设内存红线（GOMEMLIMIT）。

容器里 Go 不设 GOMEMLIMIT 很容易 OOM——因为 Go 默认按堆的 2 倍才 GC，可能超过容器 limit 被杀。

### 📈 goroutine 数量暴涨

pprof goroutine profile 看堆栈聚集在哪一行——通常是某个下游阻塞导致请求积压、channel 没有出口、缺少超时控制。

🧠 **类比**：goroutine 暴涨像"客服电话全部占线还在不停接新客户"——多半是某个环节（下游）卡住不放，新请求进来就堆着出不去。

几乎总能在 pprof 里看到几万个 goroutine 停在同一个栈帧（如某个 channel 接收或 RPC 调用），那就是元凶。

### 🏦 数据库连接池打满

排查慢查询/长事务/未释放连接（连接泄漏）；合理配置 maxOpenConns / maxIdleConns / connMaxLifetime；给 SQL 加超时；读写分离/加缓存分流。

🧠 **类比**：连接池像"银行有限的柜台"——柜台全占着（打满），要么是有人办业务超慢（慢查询/长事务），要么是有人办完不走（连接没归还）。

连接池不是越大越好，过大反而压垮 DB；要结合 DB 的最大连接数规划。

### 📝 复盘（Postmortem）应包含什么

时间线（从发生到恢复）、影响范围（多少用户/多少钱）、根因（5 Why 深挖到根）、止血与恢复过程、改进项（可落地、有 owner、有截止时间），文化上**对事不对人（blameless）**。

🧠 **类比**：复盘像"空难调查"——目的不是追责怪飞行员，而是找出系统性隐患防止再次发生（blameless）。用 5 Why 层层追问："为什么超时？→ 因为 DB 慢 → 为什么慢？→ 因为没索引 → 为什么没发现？→ 因为缺少慢查询监控…"直到根因。

好的复盘产出的是"系统性的、有主人的、有时限的改进项"，而不是"下次注意"这种空话。

### 📌 记忆卡

*   **铁律**：先止血（回滚/降级/限流/扩容）后定位根因。
*   **排查口诀**：看现象 → 分自身/下游 → 查资源 → 追变更（80% 和最近变更有关）。
*   **OOM**：pprof heap + 核对 GOMEMLIMIT；区分泄漏 vs 瞬时高峰。
*   **goroutine 暴涨**：pprof goroutine 看堆栈聚集；多因下游阻塞 + 缺超时。
*   **连接池打满**：慢查询/长事务/连接泄漏；加超时 + 缓存分流。
*   **复盘**：时间线 + 影响 + 5 Why 根因 + 改进项（有 owner 有时限）+ blameless。
