Redis 是高并发系统的"减速带"——把热点数据挡在内存里，别让流量全砸到数据库。这一篇讲透 Redis 为什么快、常见数据结构、缓存三大问题（穿透/击穿/雪崩）、缓存与数据库一致性、持久化、集群、分布式锁、大 key 热 key 治理。

### ⚡ Redis 为什么快，单线程为什么不慢

纯内存操作、高效数据结构、IO 多路复用（epoll 单线程管海量连接）、单线程避免了锁竞争和线程上下文切换。6.0+ 网络 IO 用多线程，但**命令执行仍单线程**保证原子性。

🧠 **类比**：Redis 像"一个超快的收银员用 epoll 同时盯着几百个自助结账口"——因为每单都极快（内存操作），一个人反而比多人抢一个收银机（加锁）更高效。

**副作用**：单线程意味着**一个慢命令（如 `KEYS *`、大 key 操作）会阻塞所有请求**，生产禁用这类命令。

### 🧱 常见数据结构与应用场景

*   **String**：缓存/计数/分布式锁
*   **Hash**：存对象字段
*   **List**：消息队列/最新列表
*   **Set**：去重/共同好友交集
*   **ZSet**：排行榜/延迟队列（按 score 排序）
*   **Bitmap**：签到/在线状态
*   **HyperLogLog**：海量 UV 基数估算（省内存有误差）
*   **Geo**：附近的人
*   **Stream**（5.0+）：原生消息队列，支持消费组

🧠 **类比**：ZSet 像"带分数的排行榜"，天然排序还能按分数范围查；HyperLogLog 像"用一小把沙子估算体育场人数"——不精确但极省空间。

### 🕳️ 缓存穿透、击穿、雪崩

*   **穿透**：查一个数据库里根本不存在的 key，缓存永远不命中，请求全打到 DB。解决：布隆过滤器提前拦截、缓存空值（短 TTL）。
*   **击穿**：某个**热点 key** 恰好过期的瞬间，大量并发同时打到 DB。解决：互斥锁只让一个请求重建缓存、逻辑过期（不真过期，后台异步刷新）、热点 key 永不过期。
*   **雪崩**：大量 key **同一时间**集中过期，或 Redis 整体宕机，流量瞬间压垮 DB。解决：过期时间加随机抖动、多级缓存、限流降级、Redis 集群高可用。

🧠 **类比**：穿透 = "查一个不存在的房间号，保安每次都得跑去总台确认"（布隆过滤器 = 门口先有张'有效房号名单'）；击穿 = "某明星房间的通行证刚好过期，粉丝全涌向前台"（加锁 = 只放一个人去补办）；雪崩 = "整栋楼的门禁同时失效"（错开过期时间 = 让门禁分批过期）。

**记忆口诀**：穿透查无（不存在）、击穿单点（热点过期）、雪崩批量（集中失效）。

### 🔄 缓存与数据库一致性

主流 **Cache Aside**：读时缓存没有则查 DB 回填；写时**先更新 DB，再删除缓存**（不是更新缓存）。

*   **为什么删而不是改**：避免并发写导致缓存脏值、避免写了缓存却没人读白算（懒加载）。
*   **延迟双删**：写后删一次，隔一小段再删一次，兜住"删缓存后、DB 提交前有请求读了旧值回填"的并发窗口。
*   **强一致方案**：订阅 binlog（canal）异步更新缓存、分布式锁串行化。

🧠 **类比**：缓存像"贴在门口的今日菜单副本"，后厨（DB）改了菜单就把门口副本撕掉（删缓存），下一个客人来看时发现没副本就去后厨抄一份最新的（回填）。延迟双删 = 撕完过一会再撕一次，防止有人在你撕完的空档又贴了张旧的。

缓存一致性只能做到"最终一致"，追求强一致代价极高；先问业务能否容忍短暂不一致。

### 💾 持久化 RDB vs AOF

*   **RDB**：定时快照，文件小、恢复快，但两次快照间宕机会丢数据。
*   **AOF**：记录每条写命令，丢失少（可每秒 fsync），但文件大、恢复慢。

🧠 **类比**：RDB 像"定期给内存拍全景照片"，照片之间发生的事就丢了；AOF 像"记流水账"，每笔都记，恢复时重放账本。

生产常用**混合持久化**（RDB 全量 + AOF 增量），兼顾恢复速度和数据安全。

### 🏘️ 集群方案与扩容

*   **主从复制**：读写分离基础，不高可用。
*   **哨兵 Sentinel**：监控 + 自动故障转移，主挂了选新主。
*   **Cluster**：数据分片到 16384 个槽，多主横向扩展。
*   **Proxy 分片**：Codis/twemproxy。

Cluster 扩容：把部分哈希槽迁移到新节点，客户端访问迁移中的 key 会收到 MOVED/ASK 重定向。

🧠 **类比**：主从像"主账本 + 备份抄本"；哨兵像"监工，主账本管理员倒下就立刻指定备份顶上"；Cluster 像"把 16384 个储物格分给多个管理员各管一段"，人手不够就把一部分格子交接给新管理员（迁移槽）。

Cluster 下 `MULTI` 事务/Lua 要求 key 在同一槽（用 hash tag `{}` 强制同槽）。

### 🔐 分布式锁及其坑

`SET key uniqueVal NX PX ttl` 加锁（NX 保证只有一个人拿到，PX 设过期防死锁），value 用唯一值（如 UUID），释放用 **Lua 脚本**"先比对 value 再删"防误删别人的锁。

**坑**：
*   业务没执行完锁就过期 → 用 watchdog 自动续期（Redisson 思路）。
*   主从切换锁丢失 → RedLock（多数派，有争议）。

🧠 **类比**：分布式锁像"公共会议室的门禁牌"——先到先拿（NX），牌子会自动到期免得有人忘还（PX 防死锁），还牌时先看"是不是我的牌"再还（Lua 防误还别人的）。事没办完牌快到期就续时间（watchdog）。

强一致场景（如金融扣款）别只依赖 Redis 锁，应结合 DB 唯一约束/乐观锁兜底。

### 🐘 大 key、热 key 治理

*   **大 key**（单 key 存了巨量数据）：拆分成多个小 key、用 `UNLINK` 异步删除（避免 `DEL` 阻塞）、避免大集合一次性操作。
*   **热 key**（单 key 访问量极高，打爆单节点）：本地缓存（多级缓存）、key 加随机后缀分散到多节点、读副本分流。

🧠 **类比**：大 key 像"一个塞满东西的巨型抽屉"，一开就卡住整个柜子（阻塞单线程）；热 key 像"所有人都挤同一个窗口"，得多开几个副本窗口分流。

大 key 是 Redis 单线程模型下延迟抖动的常见元凶，删除/序列化都可能阻塞。

### 🗑️ 内存淘汰策略

noeviction（默认，满了写报错）、allkeys-lru / volatile-lru（淘汰最久未用）、allkeys-lfu / volatile-lfu（淘汰最少使用，热点集中场景更优）、volatile-ttl（淘汰最快过期的）、random。

🧠 **类比**：LRU 像"整理房间时先扔最久没碰的东西"；LFU 像"先扔用得最少的东西"——如果有些东西偶尔用一次但很久没动，LRU 会误扔，LFU 更懂"高频 = 重要"。

allkeys-* 对所有 key 生效，volatile-* 只淘汰设了过期时间的 key。

### 🧵 pipeline 与 Lua 脚本

*   **pipeline**：一次性发多条命令减少网络往返（RTT），但不保证原子。
*   **Lua**：多条命令在服务端原子执行（单线程期间不被打断）。

🧠 **类比**：pipeline 像"一次把购物清单全给店员，别一样一样跑腿"（省往返）；Lua 像"这几步必须一气呵成中间不许插队"（原子）。Lua 里别写耗时逻辑（会阻塞整个 Redis）。

### 📌 记忆卡

*   **快**：纯内存 + epoll 多路复用 + 单线程无锁；慢命令会阻塞全局。
*   **缓存三问**：穿透查无（布隆/空值）、击穿单点热 key（互斥/逻辑过期）、雪崩批量过期（随机 TTL/高可用）。
*   **一致性**：Cache Aside + 先更库后删缓存 + 延迟双删 + binlog 订阅；只做最终一致。
*   **持久化**：RDB 快照恢复快、AOF 流水账丢得少；混合持久化。
*   **集群**：主从→哨兵→Cluster（16384 槽）。
*   **分布式锁**：SET NX PX + 唯一 value + Lua 释放 + watchdog 续期。
*   **大 key** 阻塞用 UNLINK；**热 key** 打散 + 多级缓存。
