工程化能力是"专家 vs 普通开发"的分水岭之一。这一篇讲透 Git 分支模型、rebase/merge、code review、CI/CD 流水线、发布策略（灰度/蓝绿/金丝雀）、可回滚可观测、以及 Go 项目工程化实践。

### 🌿 常见分支模型

*   **Git Flow**（feature/develop/release/hotfix/master）：规范但重。
*   **GitHub Flow**（主干 + feature 分支 + PR）：轻量。
*   **Trunk-Based**（几乎都在主干开发 + 特性开关）：适合高频发布 / CI 成熟团队。

🧠 **类比**：Git Flow 像"大公司严格的多级审批流程"（稳但慢）；Trunk-Based 像"小团队直接在主线上快速迭代"（快但要求测试/开关到位）。

分支越长命越难合（合并冲突/集成地狱），现代趋势是短命分支 + 频繁集成。

### 🔀 rebase vs merge

*   **merge**：保留完整历史，产生一个 merge commit，历史呈网状。
*   **rebase**：把你的提交"搬到"目标分支最新提交之后，历史线性整洁，但**改写了提交**。

**原则**：公共/共享分支**禁止 rebase**（改写他人依赖的历史会天下大乱）；本地私有分支整理提交、同步主干用 rebase。

🧠 **类比**：merge 像"如实记录两条路怎么汇合的（有个交汇点）"；rebase 像"假装我一直是接着最新进度做的，把历史捋直"。别对已经发给别人的历史 rebase（等于偷偷改了大家共同的账本）。

`git pull --rebase` 保持提交线性；出问题用 `git reflog` 找回。

### 👀 如何做 code review

**关注点**：正确性、边界条件、并发安全、错误处理是否完整、可读性/命名、测试覆盖、性能隐患、安全（注入/越权/密钥硬编码）、是否符合团队规范。

**实践**：小 PR（易审）、及时反馈、对事不对人、自动化（lint/测试）先行把关。

🧠 **类比**：code review 像"论文同行评审"——不只挑错，还帮作者提升；PR 太大像"一次交 200 页论文"没人愿意认真看。

作为专家/带人角色，review 更要引导而非直接改，帮初级理解"为什么"。

### 🏭 CI/CD 流水线阶段

拉代码 → 静态检查（lint / go vet / 安全扫描）+ 单元测试（含 -race）→ 构建产物/镜像 → 推镜像仓库 → 部署（灰度/蓝绿/金丝雀）→ 冒烟/回归测试 → 观测告警。

🧠 **类比**：CI/CD 像"食品自动化流水线 + 质检关卡"——每道工序（提交）都过安检（测试/扫描），不合格的当场拦下，合格才进下一环节直到出厂（上线）。

CI（持续集成）重在"频繁合并 + 自动验证"，CD（持续交付/部署）重在"自动化、可重复、可回滚的发布"。

### 🎏 灰度、蓝绿、金丝雀

*   **蓝绿**：准备两套完整环境（蓝=旧、绿=新），流量一次性切换，回滚快但要双份资源。
*   **金丝雀**：先放极小流量到新版本试探，逐步放量。
*   **灰度**：按用户/地域/比例分批放量。

🧠 **类比**：金丝雀发布名字来自"矿工带金丝雀下矿探毒气"——先让一小批用户"试毒"，没问题再全量。蓝绿像"两个一模一样的舞台，灯光（流量）一键切换，出问题立刻切回"。

金丝雀/灰度能在小范围暴露问题，是降低发布风险的主流手段；配合监控自动判断是否继续放量。

### ↩️ 可回滚、可观测

版本化不可变镜像、数据库变更向后兼容（先加兼容字段 → 发布 → 再清理，避免 schema 与代码不匹配）、特性开关控制、完善的监控/告警/日志 + 快速回滚预案（一键回退上一个稳定版本）。

🧠 **类比**：可回滚像"游戏存档"——每次发布都存个能秒回的档，出事就读档。数据库兼容变更像"先修好新旧都能走的路，再拆旧路"，不能一步到位把旧代码的路挖断。

数据库"先兼容后清理"（扩展-收缩模式 expand-contract）是发布安全的关键，直接改字段/删列会让回滚时旧代码崩溃。

### 🛠️ Go 项目工程化

go modules 锁版本（go.mod/go.sum）；golangci-lint 统一静态检查；表驱动测试 + `-race` + 覆盖率；接口 + mock 解耦依赖；集成测试用 testcontainers 起真实中间件。

🧠 **类比**：go.sum 像"依赖的防伪校验码"，保证每次拉到的依赖和当初一模一样（可复现构建）。表驱动测试像"一张测试用例表格，一行一个 case"，加用例只需加一行。

### 📌 记忆卡

*   **分支模型**：Git Flow 重、GitHub Flow 轻、Trunk-Based 高频；趋势是短命分支。
*   **rebase**：公共分支禁用；私有分支整理历史用。
*   **review**：小 PR + 对事不对人 + 引导思考；关注正确性/并发/错误/安全。
*   **CI/CD**：检查测试 → 构建 → 部署 → 回归 → 观测。
*   **发布**：蓝绿快切、金丝雀试探、灰度分批。
*   **发布安全**：可灰度、可回滚、DB 变更先兼容后清理（expand-contract）。
*   **Go 工程化**：go modules + golangci-lint + 表驱动测试 + -race。
