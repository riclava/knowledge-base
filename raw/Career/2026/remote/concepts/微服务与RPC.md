微服务把一个大系统拆成一堆各自负责一块业务、独立部署的小服务，服务之间靠 RPC 通信。这一篇把 gRPC、Protobuf、超时重试幂等、服务注册发现、负载均衡、链路追踪、安全、以及"该不该拆、怎么拆"这些高频问题串起来。

### 🚀 gRPC 相比 REST 的优势

gRPC 基于 **HTTP/2**（多路复用、头压缩、二进制分帧、双向流）+ **Protobuf**（紧凑二进制、强类型、代码生成）。它有四种调用模式：一元、服务端流、客户端流、双向流。

🧠 **类比**：REST/JSON 像"写信用大白话，人人能读但啰嗦"；gRPC/Protobuf 像"用双方约定的电报密码，短小高效但需先对暗号（.proto）"。HTTP/2 多路复用像"一条电话线同时并行多路通话，不用一件件排队"。

**选型**：REST 适合对外开放 API、浏览器友好、调试直观；gRPC 适合内部服务间高性能通信。gRPC 也有 grpc-gateway 转 REST 对外。

### 🔗 Protobuf 的向前/向后兼容

字段靠**编号（tag）**标识而非名字。规则：
*   新增字段用**新编号**
*   不删除/不复用旧编号（用 `reserved` 保留）
*   不改已有字段类型
*   老服务遇到不认识的新字段会**跳过**

🧠 **类比**：像"表格按列号而非列名对齐"——加新列放到末尾新列号，老程序看不懂新列就忽略，新程序遇到缺列用默认值，双方都不崩。

**proto3 的坑**：所有字段"可选"、标量默认零值，无法区分"没传"和"传了零值"（需 wrapper 或 optional 关键字解决）。

### ⏳ RPC 超时、重试、幂等三件套

*   **超时**：用 context deadline **逐级传递**，下游剩余时间随链路递减。
*   **重试**：指数退避 + 抖动（jitter）+ 最大次数上限；只对"可安全重试"的错误（网络超时、连接失败、5xx 非业务错误）重试，业务错误（如余额不足）不重试。
*   **幂等**：重试的前提是接口幂等（唯一请求 id / 幂等 token / 去重表）。

🧠 **类比**：重试像"电话没打通就再拨"，但要有礼貌——间隔越来越长（退避）、别所有人同时重拨（抖动，避免"重试风暴"打垮下游）、拨几次不通就放弃（上限）。幂等像"同一笔转账重复提交，系统认得出是同一笔，只扣一次钱"。

**重试放大效应**：A→B→C 每层各重试 3 次，C 会收到 9 倍甚至 27 倍流量，只应在**最靠近失败的一层**重试，或用重试预算（retry budget）。

### 📇 服务注册与发现

注册中心（etcd/Consul/Nacos）。服务启动时注册地址 + 定期心跳续约（租约 lease），异常下线后租约过期自动摘除；调用方拉取实例列表 + watch 变更做客户端负载均衡。

🧠 **类比**：像"外卖平台的骑手在线状态"——骑手上线登记（注册），每隔几秒上报"我还在"（心跳），断线一会儿平台就标记离线（租约过期摘除），派单系统只从在线骑手里选（发现）。

**关键细节**：健康检查（主动探测 vs 心跳上报）、优雅下线（先摘注册再停服务，避免流量打到已停实例）。

### ⚖️ 负载均衡策略与客户端/服务端 LB

*   **策略**：轮询、加权轮询、随机、最少连接、一致性哈希（会话/缓存亲和）、P2C（随机选两个挑负载低的，兼顾均衡与低开销）。
*   **客户端 LB**：调用方自己选实例（gRPC、服务网格 sidecar），少一跳、延迟低。
*   **服务端 LB**：经 Nginx/LB 集中转发，管理统一但多一跳。

🧠 **类比**：P2C 像"排队时随便看两个窗口，去人少的那个"——比全局找最优省事，又比纯随机均衡。一致性哈希像"同一个 VIP 客户总分给同一个专属客服"（缓存/会话命中率高）。

### 🔍 分布式链路追踪

OpenTelemetry / Jaeger / SkyWalking。一次请求生成全局 trace id，每个环节生成 span id 记录耗时和父子关系；通过 context + RPC metadata（HTTP header / gRPC metadata）透传。

🧠 **类比**：trace 像"快递全程物流追踪号"——一个总单号（trace id）串起每个中转站（span），每站记录几点到、几点走、上一站是谁，最后拼出完整时间线看哪段最慢。

**进阶**：采样策略（头部采样省成本 vs 尾部采样只保留出错/慢的更精准）；跨异步边界（MQ）要把 trace 信息塞进消息头才能续接。

### 🔐 服务间通信安全

mTLS 双向证书认证、服务身份（SPIFFE）、token/JWT 鉴权、服务网格统一策略、网络分段隔离。

🧠 **类比**：mTLS 像"双方都要出示工作证"——不只是我确认你（单向 TLS），你也要确认我，冒充者进不来。

内网也不应"默认信任"（**零信任**理念），东西向流量同样鉴权。

### ✂️ 微服务拆分原则，拆太细的代价

**原则**：按业务领域边界（DDD 限界上下文）、变更频率、团队边界拆；追求高内聚低耦合，一个服务拥有自己的数据（不共享库）。

**拆太细的代价**：分布式复杂度飙升（原本函数调用变网络调用）、跨服务事务困难、链路变长排障难、运维/部署成本高、性能损耗。

🧠 **类比**：拆服务像"分部门"——按职能分（销售/财务）而不是随便切。分太细就像"每个人一个部门"，开会（网络调用）成本、扯皮（一致性）成本全上来了。

**建议**：不确定边界时先"模块化单体"，边界清晰后再拆；避免"分布式单体"（拆了但强耦合、必须一起发布）。

### 🧰 补充：序列化选型与 API 网关

**序列化协议选型**：Protobuf/Thrift（二进制、快、需 schema）、JSON（可读、通用、慢）、MessagePack、Avro（大数据）。内部服务选二进制，对外/调试选 JSON。

**API 网关职责**：统一入口做路由转发、鉴权认证、限流熔断、协议转换、日志监控、灰度分流。类比：API 网关像"公司前台 + 保安"——所有访客统一从这进，验身份、登记、指路、拦截可疑人员，内部各部门（微服务）不用各自设岗。

### 📌 记忆卡

*   **gRPC**：HTTP/2 + Protobuf；四种流模式；内部用 gRPC 对外用 REST。
*   **Protobuf 兼容**：靠字段编号；新增用新号、旧号 reserved、不改类型。
*   **RPC 三件套**：超时逐级递减、重试要退避+抖动+上限+只重可重试的、幂等兜底。
*   **重试放大**：只在最靠近失败一层重试。
*   **注册发现**：注册 + 心跳续约 + 租约过期摘除 + 优雅下线。
*   **负载均衡**：P2C 兼顾均衡与开销；一致性哈希做亲和。
*   **链路追踪**：trace id 全局 + span 记耗时；跨 MQ 要透传。
*   **拆分**：按 DDD 边界；能不拆先别拆；忌分布式单体。
