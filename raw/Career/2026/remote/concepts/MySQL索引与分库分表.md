MySQL（InnoDB）是绝大多数后端系统的权威数据源。这一篇从 B+ 树索引原理讲起，覆盖聚簇/二级索引、最左前缀、慢查询优化、索引失效、事务隔离与 MVCC、三大日志与两阶段提交，最后到分库分表的方案与难题。

### 🌳 InnoDB 为什么用 B+ 树

B+ 树"矮胖"：非叶子节点只存 key 不存数据，一个节点能放更多 key（扇出大），树高低（一般 3~4 层就能存千万级），查一次只需几次磁盘 IO。叶子节点用**双向链表**串联，范围查询、排序极友好。

*   **哈希**：只支持等值查询，不支持范围/排序。
*   **红黑树**：二叉树太高，磁盘 IO 次数多。
*   **B 树**：数据分散在每层，范围查询要跨层，且单节点存数据导致扇出小、树更高。

🧠 **类比**：B+ 树像"图书馆的分级索引卡"——目录页（非叶子）只写"哪类书在哪个区"（只存 key，一页能写很多），真正的书都在最底层书架（叶子）且按顺序排成一排（链表），找连续一批书（范围查询）顺着书架走就行。

磁盘 IO 是瓶颈，B+ 树的设计核心就是"用尽量少的 IO 定位数据"。

### 📂 聚簇索引、二级索引与回表

*   **聚簇索引（主键索引）**：叶子节点直接存**整行数据**，数据按主键物理有序。
*   **二级索引（普通索引）**：叶子节点只存**主键值**。
*   **回表**：用二级索引查到主键后，还得拿主键去聚簇索引再查一次取完整行——两次查找。**覆盖索引**（要查的字段都在索引里）可避免回表。

🧠 **类比**：聚簇索引像"按身份证号排列的完整档案柜，找到号就拿到全部资料"；二级索引像"按姓名排的小卡片，卡片上只写了身份证号"，想看完整资料还得拿号去大档案柜再翻一次（回表）。只想知道身份证号，看小卡片就够了（覆盖索引）。

主键不宜过长（二级索引都存主键）；建议用自增主键（顺序插入，避免页分裂）。

### 🔤 最左前缀原则与联合索引设计

联合索引 `(a,b,c)` 相当于建了 `(a)`、`(a,b)`、`(a,b,c)` 三个前缀索引；查询必须从最左列开始连续匹配才能用上索引，跳过 a 直接查 b 用不上。

**设计**：等值查询列放前、范围查询列放后（范围列之后的列无法用索引）、高区分度列靠前、尽量覆盖查询字段。

🧠 **类比**：联合索引像"电话簿按'姓→名→中间名'排序"——你知道姓能快速定位，知道姓+名更精确；但只知道名不知道姓，电话簿就帮不上忙（不满足最左前缀）。

例：`WHERE a=1 AND b>5 AND c=3`，索引 `(a,b,c)` 只能用到 a 和 b（b 是范围，c 用不上）；调成 `(a,c,b)` 可让 c 也生效。

### 🐢 慢查询分析与优化

开慢查询日志定位 SQL → `EXPLAIN` 分析执行计划，重点看：
*   **type**（连接类型，全表 ALL 最差、const/ref/range 较好）
*   **key**（实际用的索引）
*   **rows**（预估扫描行数）
*   **Extra**（`Using filesort` 额外排序、`Using temporary` 临时表、`Using index` 覆盖索引好信号）

**优化**：加合适索引、避免索引失效、改写 SQL、分页优化、必要时冗余/反范式。

🧠 **类比**：`EXPLAIN` 像"导航出发前预览路线"——告诉你走哪条路（索引）、经过多少路口（rows）、有没有堵车路段（filesort/临时表）。type=ALL 就是"没走高速全程走小路"。

### 🚫 索引失效的常见情况

索引列上用函数/运算（`WHERE YEAR(t)=2024`）、隐式类型转换（字符串列传数字）、`LIKE '%x'` 前置通配、`OR` 连接非索引列、不满足最左前缀、范围查询后的列、`!=`/`NOT IN`/`IS NOT NULL` 部分场景。

🧠 **类比**：索引像"字典的偏旁部首检字表"——按部首查很快；但如果你把字先"变形"了（加函数），或只知道字的后半部分（`%x`），检字表就用不上，只能一页页翻（全表扫描）。

改写示例：`WHERE YEAR(create_time)=2024` → `WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'` 就能用索引。

### 🔐 事务隔离级别、MVCC 与幻读

**四级**：读未提交（脏读）、读已提交（RC，防脏读）、可重复读（RR，默认，防不可重复读）、串行化（全防但慢）。

*   **MVCC**：通过 undo log 版本链 + ReadView（一致性视图）实现"快照读"——不加锁读到某个时间点的版本。
*   **幻读**：RR 下用**间隙锁（gap lock）+ 记录锁 = next-key lock** 锁住范围，防止别人在区间插入新行。

🧠 **类比**：MVCC 像"给数据拍快照"——你的事务开始时拍了张照，之后一直看这张照片（快照读），别人怎么改都不影响你看到的。间隙锁像"不仅锁住已有座位，连座位之间的空隙也占住"，防止有人在你查询的范围里塞新座位（幻读）。

快照读（普通 SELECT）靠 MVCC，当前读（`SELECT ... FOR UPDATE`/`LOCK IN SHARE MODE`/更新）靠加锁。

### 📜 redo log、undo log、binlog 与两阶段提交

*   **redo log**：InnoDB 物理日志，记录"某页做了什么修改"，用于崩溃恢复（持久性），循环写。
*   **undo log**：记录数据旧版本，用于事务回滚 + MVCC 快照读。
*   **binlog**：Server 层逻辑日志，记录 SQL/行变更，用于主从复制、数据恢复，追加写。
*   **两阶段提交**：redo prepare → 写 binlog → redo commit，保证崩溃后 redo 和 binlog 状态一致。

🧠 **类比**：redo 像"施工时的'已完成动作'便签"，断电后照便签把没落实的补上（恢复）；undo 像"每步操作前拍的'改之前长啥样'照片"，撤销就照片还原；binlog 像"官方施工记录"给从库照着复刻。两阶段提交像"先预备签字、等官方记录也写好、再正式签字"，确保两本账一致。

redo 保证 crash-safe，binlog 保证复制，两者用 2PC 绑定一致，是 MySQL 可靠性的核心。

### 🪓 分库分表方案与拆分键

*   **垂直拆分**：按业务把不同表拆到不同库（用户库/订单库）、把大表的冷字段拆出去。
*   **水平拆分**：同一张表按 range（时间/id 段）或 hash（取模/一致性哈希）把行分散到多张表/多个库。
*   **拆分键（sharding key）选择**：高频查询维度（如订单按 user_id）、分布均匀避免热点、尽量让常见查询能路由到单片。

🧠 **类比**：垂直拆分像"把杂货铺按品类分成多家专卖店"；水平拆分像"一家店客人太多，按'姓氏首字母'开多家分店分流"。拆分键选不好（如按时间且都查最新）会导致新店挤爆、老店闲置（热点）。

中间件 ShardingSphere / MyCat / 自研；**能不拆就不拆**（先读写分离、加缓存、归档冷数据），单表通常千万级、超大才考虑分表。

### 🧩 分库分表后的难题

*   **分页**：`ORDER BY` 全局排序需从每个分片取 `offset+limit` 再归并，深分页代价大（优化：禁止深翻 / 二次查询法 / 用游标）。
*   **跨库 join**：避免，改为多次查询 + 应用层拼装，或冗余字段/宽表。
*   **全局 id**：雪花算法（趋势递增）、号段模式（DB 批量发号）、Redis incr。
*   **扩容**：一致性哈希 + 虚拟节点减少迁移量、双写迁移、按 2 的倍数扩容降低 rehash 范围。

🧠 **类比**：分库分表后像"账本被拆到多个分店"——查排行榜（分页排序）得把每家分店的前几名都拿来再汇总排（归并）；想把两类数据关联（join）就得跑几家店手动对（应用层拼装）；发会员号（全局 id）不能各店各发免得撞号，得统一发号机。

### 🔄 主从复制与主从延迟

**原理**：主库写 binlog → dump 线程发给从库 → 从库 IO 线程写 relay log → SQL 线程重放。有半同步、并行复制（多线程重放）优化。

**延迟处理**：对一致性要求高的读走主库、缓存标记刚写过的数据、并行复制、监控 `Seconds_Behind_Master`。

🧠 **类比**：主从像"老师在黑板上写（主库），学生抄笔记（从库）"——学生抄得慢就有延迟。刚讲的重点（刚写的数据）想立刻确认就直接问老师（读主库）。"写后立即读"读到旧数据是主从延迟的典型坑。

### 📖 深分页为什么慢，如何优化

`LIMIT 1000000,20` 时 MySQL 要先扫描并**丢弃**前 100 万行才返回 20 行，越翻越慢。

*   **游标法**：`WHERE id > last_max_id ORDER BY id LIMIT 20`（记住上次位置）。
*   **延迟关联**：先用覆盖索引查出主键再回表取数据。

🧠 **类比**：深分页像"看一本书要翻到第 5 万页，你从第一页一页页数过去（扫描丢弃）"，太蠢。游标法像"用书签记住上次看到哪，下次直接从书签翻起"。记住两个关键词——**游标 + 延迟关联**。

### 🧰 补充：范式/反范式，乐观锁/悲观锁

**范式 vs 反范式**：范式减少冗余（数据一致、省空间），反范式增加冗余（用空间/一致性换查询性能，减少 join）。高并发读场景常适度反范式。

**乐观锁 vs 悲观锁**：悲观锁先加锁再操作（`SELECT ... FOR UPDATE`），假设一定有冲突；乐观锁不加锁，更新时用版本号/CAS 校验（`UPDATE ... WHERE version=?`），假设很少冲突。类比：悲观锁像"进厕所先反锁门"，乐观锁像"不锁门，出来时确认没人趁机进过（版本没变）才算数"。读多写少用乐观锁，写多冲突频繁用悲观锁。

### 📌 记忆卡

*   **B+ 树**：矮胖、非叶只存 key、叶子链表；少 IO 定位 + 范围友好。
*   **回表**：二级索引查到主键再查聚簇；覆盖索引免回表。
*   **最左前缀**：从最左列连续匹配；范围列之后失效。
*   **EXPLAIN**：看 type/key/rows/Extra。
*   **索引失效**：函数运算、隐式转换、前置 %、不满足最左前缀。
*   **MVCC**：undo 版本链 + ReadView 快照读；RR 用 next-key lock 防幻读。
*   **三大日志**：redo（崩溃恢复/物理）、undo（回滚/MVCC）、binlog（复制/逻辑）；2PC 保一致。
*   **分库分表**：能不拆先别拆；痛点在分页/join/全局 id/扩容。
*   **深分页**：游标 + 延迟关联。
