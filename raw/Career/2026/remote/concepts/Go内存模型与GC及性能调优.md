Go 之所以既能写得爽又跑得快，很大程度靠它的**自动内存管理 + 低延迟 GC**。这一篇把三色标记 GC、写屏障、GC 触发时机、以及一整套 pprof 性能调优方法论讲透，最后落到几个专家级排障场景（内存泄漏、CPU 打满、P99 突刺）。

### 🎨 GC 算法：三色标记 + 写屏障

Go 用的是**并发标记清除**（非分代、非压缩）。三色抽象：

*   **白色** = 未访问（待回收）
*   **灰色** = 已发现待扫描
*   **黑色** = 已扫描完毕

从根（栈、全局变量）出发，灰变黑、把它引用的白变灰，直到没有灰色，剩下的白色回收。

**难点**：标记和程序并发跑，程序可能"黑对象新指向一个白对象、又断开灰对象到它的引用"，导致白对象被误回收。**混合写屏障**（Go 1.8+）在指针写操作时做记录，保证不漏标，把 STW 压到亚毫秒级。

🧠 **类比**：三色标记像"给房子里的东西贴标签大扫除"——从门口（根）开始，看得见的先贴灰，检查完贴黑，同时把它连着的东西也贴灰；最后还是白的（没人用的）就扔掉。写屏障像"打扫时有人偷偷搬东西，门口安个记录员，凡是搬动都登记，保证不误扔还在用的东西"。

STW 只发生在标记开始（开启写屏障）和标记结束（关闭写屏障）两个极短点，其余并发进行。Go GC 优化目标是**低延迟**而非最高吞吐，所以不做分代/压缩。

### 🔥 GC 何时触发：GOGC 与 GOMEMLIMIT

*   **GOGC（默认 100）**：当**堆增长到上次 GC 后存活对象的 2 倍**（100%）时触发。调大 → GC 更少但内存占用高；调小 → 更省内存但 GC 更频繁。
*   **GOMEMLIMIT（1.19+）**：软内存上限，接近上限时更激进地 GC，防容器 OOM。

🧠 **类比**：GOGC 像"垃圾桶多满才倒"的阈值——设 100 就是"攒到上次清完后的 1 倍新垃圾就倒一次"。GOMEMLIMIT 像"房间总容量红线"，快到红线就疯狂清理别撑爆。

**容器实践**：务必设 GOMEMLIMIT ≈ 容器内存 limit 的 ~90%，并配合 GOMAXPROCS 适配 CPU limit（否则默认读宿主机核数，调度/GC 失衡）。

### 🕳️ 排查内存泄漏（线上内存持续上涨）

用 `pprof heap`：`inuse_space` 看当前占用、`alloc_space` 看累计分配；抓多次快照对比增长点，定位分配调用栈。

**常见根因**：goroutine 泄漏（每个 goroutine 持有对象）、全局 map/缓存只增不减、slice/substring 引用大底层数组、资源未 Close、`time.Ticker`/`Timer` 未 Stop、cgo 内存。

🧠 **类比**：内存泄漏像"水池只进不出"，先找哪个水龙头一直开着（哪段代码持续分配又不释放）。pprof 快照对比像"每隔一会拍张水位照片，看哪块区域水位一直涨"。

**要区分三种"涨"**：真泄漏（对象一直被引用）、GC 未及时（临时高峰）、RSS 未归还 OS（Go 会缓存内存不马上还，属正常）。

### 🌋 CPU 使用率高如何定位热点

`pprof cpu` 采样 30s + 火焰图（`go tool pprof -http=:8080`），火焰图越宽的函数占 CPU 越多。

**关注点**：频繁分配触发 GC（GC 占 CPU）、锁竞争（mutex/block profile）、序列化/正则/加解密热点、无谓的反射。

🧠 **类比**：火焰图像"从上往下看的一叠账单"，哪一层横条最长，钱（CPU）就花在那；顺着往下钻就能找到最烧钱的那笔。

结合系统层面看：`us` 高看应用代码、`sy` 高看系统调用（如频繁 write/网络）、`GC` 高看分配。

### 📉 如何降低 GC 压力

核心是**减少堆分配**：预分配 slice/map 的 cap、复用对象（sync.Pool）、小对象值传递、避免不必要的指针和 interface 装箱、减少逃逸。

🧠 **类比**：GC 压力大像"清洁工被海量小垃圾累垮"。少产生垃圾（少分配）、能重复用的碗别每次换新的（对象复用），清洁工就轻松了。

`GODEBUG=gctrace=1` 打印每轮 GC 的耗时、频率、堆大小，是判断"GC 是不是瓶颈"的第一手数据。

### 🤝 内存模型中的 happens-before

规定跨 goroutine 的内存操作可见性顺序。没有同步原语建立 happens-before，就不能假设一个 goroutine 的写对另一个可见（编译器/CPU 可能重排）。

**建立方式**：channel 收发、Mutex Lock/Unlock、Once.Do、atomic、WaitGroup、go 语句启动。

🧠 **类比**：像"接力交棒"——只有棒交到手上，你才确定上一棒跑完了。核心记忆：**共享数据的可见性不是"写了就能看到"，而是"同步过才能看到"**。

### 🩺 pprof 的几种 profile 类型

*   **cpu**：热点函数
*   **heap**：内存分配/占用
*   **goroutine**：协程堆栈与数量
*   **block**：阻塞在同步原语的时间
*   **mutex**：锁竞争
*   **trace**：调度/GC/网络的时间线全景

🧠 **类比**：pprof 像"体检套餐"——cpu 查心率（谁最累）、heap 查体重（内存胖在哪）、goroutine 查人数（是否人满为患）、mutex/block 查排队（哪堵）、trace 是"全天行为录像"。

生产环境用 `net/http/pprof` 暴露端点（注意鉴权，别裸奔）；trace 用 `runtime/trace` 采集后 `go tool trace` 看。

### ⚡ P99 延迟突刺如何定位

**排查维度**：GC STW / mark assist（分配太快被拉去帮忙标记）、锁竞争、goroutine 暴涨导致调度排队、下游依赖（DB/缓存/RPC）抖动、连接池耗尽、GC 触发点与流量高峰重叠。

**工具**：`runtime/trace` 看 STW 和调度延迟、metrics（GC pause、goroutine 数、P99）、分布式链路追踪定位是本服务还是下游耗时。

🧠 **类比**：P99 突刺像"高速公路平时畅通但偶尔堵一下"——可能是清障车临时封道（GC STW）、收费站排队（锁/连接池）、前方路段（下游）事故。要装"行车记录仪（trace/链路）"才能还原那一瞬间。

**隐蔽元凶——mark assist**：分配速率过高时，分配的 goroutine 会被强制"打工帮 GC 标记"，表现为个别请求莫名变慢。降低分配速率或调 GOGC 缓解。

### 📚 栈 vs 堆，goroutine 栈如何增长

*   **栈**：函数私有、随返回自动回收、访问快。
*   **堆**：跨函数共享、GC 管理。
*   goroutine 初始栈仅 **2KB**，不够时**连续栈扩容**（拷贝到更大栈），也可缩容。

🧠 **类比**：goroutine 栈像"可伸缩的抽屉"，初始很小（2KB，所以能开百万协程），东西多了自动换更大的抽屉并搬过去。这也是 Go 能轻量开海量协程的原因（对比线程默认几 MB 栈）。

### 🏁 benchmark 怎么写，如何防被优化掉

`func BenchmarkXxx(b *testing.B)` + `for i:=0;i<b.N;i++`；`b.ReportAllocs()` 看分配；结果赋给包级变量或用 `runtime.KeepAlive` 防止被编译器优化消除。

🧠 **类比**：benchmark 像"跑步机测心率"，`b.N` 是系统自动决定的"跑多少步才测得准"。

### 📌 记忆卡

*   **GC**：三色标记 + 混合写屏障 → 亚毫秒 STW；非分代非压缩，优化低延迟。
*   **触发**：堆到存活对象 2 倍（GOGC=100）；容器设 GOMEMLIMIT。
*   **内存泄漏**：pprof heap 多次快照对比；区分真泄漏/GC 未及时/RSS 未归还。
*   **CPU 热点**：pprof cpu + 火焰图，越宽越费 CPU。
*   **降 GC 压力**：预分配 + 复用 + 减少逃逸；gctrace=1 观察。
*   **P99 突刺**：GC STW / mark assist / 锁 / 下游抖动；trace 还原。
*   **goroutine 栈**：2KB 起步，连续栈扩容。
