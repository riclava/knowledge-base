Elasticsearch（ES）是后端系统里最常用的全文检索与聚合分析引擎。它的核心是**倒排索引**。这一篇讲透倒排原理、正排 doc values、写入近实时机制、分片副本设计、深分页优化、与 MySQL 的数据同步、以及深度调优点。

### 🔎 倒排索引原理

正常思路是"文档 → 包含哪些词"（正排），倒排反过来："词 → 出现在哪些文档"。分词器把文本拆成词项（term），建立 词项 → 文档列表（posting list）的映射；查询时找 term 命中的文档并按相关性打分（BM25）。

🧠 **类比**：正排像"每本书的目录（这本书讲了啥）"；倒排像"书末的'索引'——某个关键词出现在哪几页"。搜"分布式"时直接翻索引找到所有提到它的页，而不是把每本书从头读一遍。

倒排索引是所有搜索引擎的基础；分词质量直接决定搜索效果（中文分词更复杂）。

### 📊 正排（doc values）vs 倒排的用途

倒排用于"根据词找文档"（全文检索）；doc values 是列式存储的正排，用于排序、聚合、脚本计算（根据文档找字段值）。

🧠 **类比**：倒排回答"哪些文章提到了 Go"（搜索）；doc values 回答"把这些文章按发布时间排序 / 按作者分组统计"（排序聚合）。两个方向的需求要两套结构。

### ⏱️ ES 写入流程与近实时（NRT）

写入先进内存 buffer + 写 translog（防丢）→ 每隔 refresh_interval（默认 1s）refresh 生成一个可搜索的 segment（所以"近实时"，写完约 1s 才能搜到）→ 定期 flush 把 segment 落盘并清 translog → 后台 merge 合并小 segment。

🧠 **类比**：像"草稿 → 定时誊抄成可查阅的卡片（refresh，1s 一次所以近实时）→ 装订归档（flush）→ 定期把零散卡片合并成大卡片（merge）"。translog 像"随手记的流水账"，防止誊抄前断电丢数据。

调大 refresh_interval 能提升写入吞吐（代价是搜索延迟变大）；批量导入时可临时关闭 refresh。

### 🧩 分片（shard）与副本（replica）设计

主分片数**创建后不可改**（要改只能 reindex），决定数据如何拆分和最大扩展能力；副本可动态增减，提升读吞吐和高可用（主挂了副本顶上）。

🧠 **类比**：主分片像"把一个大仓库的货分到 N 个库房"（几个库房一开始就定死）；副本像"每个库房再复制几份备份仓"，既能分担查询又能容灾。

分片不是越多越好——每个分片有固定开销，过多小分片浪费资源；单分片建议控制在几十 GB。

### 📖 深分页优化

`from + size` 深分页时，每个分片都要取 `from+size` 条再汇总排序，内存和 CPU 暴涨（默认限制 1 万）。用 `search_after`（基于上一页最后一条的排序值做游标）翻页，或 `scroll`（快照式批量导出，适合导数据不适合实时翻页）。

🧠 **类比**：和 MySQL 深分页一个道理——别"翻到第 1000 页"，而是"记住上页最后一条，从它往后取"（search_after = 游标）。

面向用户的翻页用 search_after，后台全量导出用 scroll 或 PIT（point in time）。

### 🔄 ES 与 MySQL 数据同步

双写（简单但一致性差）、订阅 binlog（canal）异步同步（推荐，解耦）、Logstash/自研同步服务。关注顺序、幂等、延迟、失败重试。

🧠 **类比**：ES 是 MySQL 数据的"可搜索副本"，靠"监听 MySQL 的改动日志（binlog）自动跟着更新"最省心，避免业务代码里到处双写。

MySQL 做权威数据源（source of truth），ES 只做检索加速，不一致时以 MySQL 为准。

### 🎛️ ES 深度调优点

mapping 精简（不需分词的用 keyword、不需要的关掉 norms/_source 部分字段）、批量 bulk 写入、合理 refresh_interval、冷热数据分层（热节点 SSD、冷节点廉价存储）、定期 force merge 减少 segment。

🧠 **类比**：keyword vs text 像"这字段是拿来精确匹配（身份证号 = keyword）还是全文搜索（文章内容 = text）"，选错会又占空间又搜不准。

### 📌 记忆卡

*   **倒排**：词 → 文档列表；全文检索基础；分词决定效果。
*   **doc values**：列式正排，用于排序/聚合。
*   **近实时**：buffer + translog → refresh(1s) 生成 segment → flush → merge。
*   **分片副本**：主分片建后不可改，副本可增减、提升读与高可用。
*   **深分页**：search_after（用户翻页）/ scroll、PIT（导出）。
*   **同步**：canal 订阅 binlog；MySQL 为准。
