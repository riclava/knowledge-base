系统设计题是专家面试的重头戏，考的不是背答案，而是**在约束下做权衡的能力**。这一篇先给一套万能方法论，再逐个拆解短链、计数、Feed、延迟队列、IM、订单支付、架构演进这些经典场景。

### 🧭 通用方法论（万能框架）

**需求澄清 → 容量估算（QPS/存储/带宽）→ 核心接口与数据模型 → 存储选型 → 缓存与一致性 → 高可用与水平扩展 → 瓶颈分析与权衡取舍**。

🧠 **记忆**：先问清楚要什么（需求）→ 算多大（容量）→ 怎么存（模型/存储）→ 怎么快（缓存）→ 怎么稳（高可用）→ 哪里会挂（瓶颈）。

系统设计没有标准答案，面试官看的是你有没有主动澄清需求、有没有量化估算、有没有说清每个选择的 tradeoff。

### 🔗 短链系统

发号（雪花/号段）→ base62 编码成短码；读多写少 → Redis 缓存热点 + 布隆过滤器挡无效码；访问用 302 跳转；防刷限流；点击统计异步化（MQ）。

🧠 **类比**：短链像"给长网址发一个简短的会员号"，核心是"发号 + 编码 + 高速查表跳转"。

### 👍 高并发计数/点赞系统

Redis 原子自增扛写 + 定时/批量异步落库；热点计数打散（分桶累加后合并）；用户维度去重防重复点赞；最终一致。

🧠 **类比**：像"演唱会实时人数计数器"——不可能每进一人就写一次数据库，先在内存快速累加，定期同步到账本（DB）。

### 📰 Feed / 时间线系统

*   **推模式（写扩散）**：发帖时写入所有粉丝的收件箱，读快写重，适合普通用户。
*   **拉模式（读扩散）**：读时实时聚合关注者的帖子，写轻读重，适合大 V。
*   **推拉结合**：普通用户推、大 V 拉。

🧠 **类比**：推模式像"群发短信到每个粉丝信箱"（发一次累死，但粉丝看信箱很快）；拉模式像"粉丝自己去每个关注对象主页翻"（大 V 有千万粉就不能推）。微博这种就用推拉结合。

这是"写扩散 vs 读扩散"的经典权衡，答题要点出"根据用户粉丝量分层处理"。

### ⏰ 分布式延迟队列 / 定时任务

方案：Redis ZSet（score = 执行时间戳，轮询到期的）、时间轮（高效管理海量定时器）、RocketMQ 延迟级别、DB 扫表（简单但不精准）。关注幂等、避免重复触发、水平扩展下的任务分片。

🧠 **类比**：ZSet 延迟队列像"按'该叫醒的时间'排序的闹钟列表"，轮询看有没有到点的；时间轮像"钟表的齿轮"，指针转到哪格就触发那格的任务，管理海量定时器很省。

### 💬 IM / 消息推送系统

WebSocket 长连接网关（维护在线连接）+ 路由层（用户 → 连接在哪台机器）+ 在线状态管理 + 消息可靠投递（ack + 重试 + 去重 + 离线存储）+ 群消息扩散。

🧠 **类比**：IM 像"邮局 + 实时对讲机"——在线的直接对讲（长连接推送），不在线的存到信箱（离线消息），上线后补发。核心难点是"找到对方现在连在哪台服务器"（路由）和"消息不丢不重"。

### 🛒 订单/支付系统核心链路

幂等下单（防重复提交）、库存扣减防超卖（Redis Lua / DB 乐观锁）、支付回调对账、超时未支付关单（延迟队列）、状态机管理订单状态流转、分布式事务/最终一致保证钱货一致。

🧠 **类比**：订单系统像"严谨的收银流程"——每一步状态明确（下单 → 待支付 → 已支付 → 发货，状态机），钱和货必须对得上账（对账），同一笔不能重复扣款（幂等）。

钱相关系统强调"幂等 + 对账 + 状态机 + 最终一致"，可靠性优先于性能。

### 🏗️ 亿级流量下的系统重构 / 架构演进

演进路径：单体 → 垂直拆分 → 服务化/微服务 → 中台化。方法：识别瓶颈分阶段迁移、新旧并行双跑对比结果、灰度切流、数据迁移用双写 + 校验 + 回滚预案，保证不停机平滑过渡（绞杀者模式 strangler）。

🧠 **类比**：重构大系统像"给正在飞的飞机换引擎"——不能停机，得一个引擎一个引擎地换（分阶段），换的时候新旧引擎并行验证（双跑），确认新的没问题再拆旧的（绞杀者）。

核心原则——**小步快跑、可灰度、可回滚、数据可校验**，绝不"推倒重来一把梭"。
