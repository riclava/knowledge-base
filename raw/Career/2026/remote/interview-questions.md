## 1. Go 语言底层与运行时

**1.1 ⭐⭐ GMP 调度模型是怎样的？P 的作用是什么？**
- 关键点：G(goroutine，任务)、M(Machine，OS 线程)、P(Processor，调度上下文/许可证)。P 数量默认 `GOMAXPROCS`（通常=CPU 核数）。
- M 必须持有一个 P 才能执行 G；P 有本地可运行队列（runq，容量 256），减少全局锁竞争。
- 调度来源优先级：本地队列 → 全局队列 → netpoll → work-stealing（空闲 P 从别的 P 偷一半 G）。
- 🧠 类比：把 P 想成「工位」，M 是「工人」，G 是「待办任务」。工位数量固定（GOMAXPROCS），工人再多也只有拿到工位才能干活；每个工位有自己的任务篮（本地队列），自己篮子空了就去别人篮子里"偷"一半任务，避免有人闲死有人累死。
- 延伸：为什么引入 P？早期只有 GM，所有 G 抢一把全局锁，锁竞争严重、无本地缓存。加了 P 就有了「每工位一个任务篮」，大幅降低锁争用。

**1.2 ⭐⭐⭐ goroutine 发生阻塞（系统调用 / channel / 网络）时调度器如何处理？**
- 阻塞式系统调用（如读文件）：M 会被系统调用卡住，于是 M 和 P 解绑（handoff），P 交给另一个空闲/新建 M 继续跑其他 G，避免整个工位闲置。
- 网络 IO：交给 netpoller（底层 epoll/kqueue）托管，G 被挂起但**不占用 M**，数据就绪后 G 重新入队。
- channel 阻塞：G 进入 channel 的等待队列（sudog），M 立刻去跑别的 G，线程不阻塞。
- 🧠 类比：工人（M）去办一件"必须现场排队"的事（阻塞系统调用），就把工位（P）让给另一个工人接着干别的活；而网络 IO 像"取号后回来等叫号"，人不用一直站队里（不占 M），号到了再回来。
- 延伸：区分记忆——**阻塞系统调用会占用 M**（所以可能新建线程），**网络 IO 走 netpoller 不占 M**（Go 高并发网络的关键）。

**1.3 ⭐⭐ 抢占式调度是如何实现的？sysmon 的作用？**
- 早期是协作式抢占（只在函数调用的栈检查点让出），死循环无函数调用会饿死其他 G；Go 1.14 起改为**基于信号的异步抢占**，运行时给线程发信号强制让出。
- sysmon 是后台监控线程（不绑 P 独立运行）：抢占运行超 10ms 的 G、回收陷入系统调用过久的 P、定期触发 GC、执行 netpoll。
- 🧠 类比：sysmon 像"车间巡检员"，看到某个工人霸占工位太久（>10ms），就拍他肩膀强制换人，保证任务轮转公平。
- 延伸：1.14 前经典 bug——`for {}` 空循环能把某个 P 卡死导致 GC 无法进行（STW 等不到抢占点）。

**1.4 ⭐ slice 和 array 的区别？slice 扩容机制？**
- array 是值类型、长度固定，赋值/传参是整份拷贝；slice 是引用视图，底层是结构体 `{ptr 指向底层数组, len 长度, cap 容量}`。
- 扩容规则（简化）：期望容量 > 2*旧cap 直接用期望值；旧 cap < 256 翻倍；≥256 每次约 1.25 倍平滑增长。扩容会分配**新数组并拷贝**，原 slice 与新 slice 脱钩。
- 🧠 类比：array 像"固定格子的冰格"，大小定死；slice 像"指向某段冰格的便利贴"，写着起点、已用几格、总共几格。扩容就像格子不够了换个更大的冰盒并把冰倒过去。
- 延伸：共享底层数组的坑——`b := a[1:3]` 后对 b 的 append 若未超 cap 会**改到 a 的数据**；截取大数组的小 slice 会导致大数组无法 GC（内存泄漏）。

**1.5 ⭐⭐ map 的底层实现？为什么无序？能并发读写吗？**
- 结构：hmap + 若干 bucket（bmap），每个 bucket 存 8 个 key/value，放不下用溢出桶（overflow）链接；负载因子超阈值触发**渐进式扩容**（增量搬迁 oldbuckets，不一次性搬完）。
- 无序：遍历时随机选起始 bucket 和槽位，故意防止你依赖顺序。
- 并发读写会触发 `fatal error: concurrent map read and map write`（连 recover 都拦不住），必须用 `sync.Map` 或加锁。
- 🧠 类比：map 像"一排储物柜（bucket），每个柜子 8 个格子"。key 经哈希决定进哪个柜子，柜子满了就挂个"加长抽屉"（溢出桶）。搬家（扩容）不是一次搬完，而是每次访问顺手搬一点（渐进式），避免一次卡顿。
- 延伸：`sync.Map` 适合"读多写少 / key 相对固定"，内部用 read + dirty 双 map 减少锁；写多场景不如"分片锁 map"。

**1.6 ⭐⭐ interface 的底层结构？iface 和 eface 区别？nil interface 陷阱？**
- eface（空接口 `interface{}`/`any`）：`{_type 类型指针, data 数据指针}`；iface（带方法的接口）：`{itab, data}`，itab 里含具体类型信息 + 方法地址表。
- nil 陷阱：接口只有**类型和值都为 nil** 时才 `== nil`。若把一个 `(*T)(nil)` 赋给接口，接口有类型信息，`i == nil` 为 false。
- 🧠 类比：接口像"带标签的盒子"——标签写"里面装的是什么类型"，盒子里放"数据地址"。哪怕盒子里是空的（data=nil），只要贴了标签（有类型），它就不等于"完全的空盒子（nil）"。
- 延伸：经典坑——函数返回 `error` 接口，内部 `var err *MyErr = nil; return err`，调用方 `if err != nil` 恒成立。修复：返回时判断或直接 `return nil`。

**1.7 ⭐ defer 的执行顺序与性能？1.14 后有何优化？**
- 多个 defer 按 **LIFO（后进先出）** 执行；defer 的**参数在注册那一刻就求值**（不是执行时）。
- 1.13 引入 defer 池化、1.14 引入 open-coded defer（编译期直接内联展开），开销接近普通函数调用。
- 🧠 类比：defer 像"叠盘子"——最后叠上去的盘子最先被拿走（LIFO）。参数求值像"贴便签时就把当时的值写死了"，后面变量再变也不影响便签内容。
- 延伸：循环里 `defer` 会累积到函数结束才释放（如循环内 defer file.Close 导致文件句柄堆积），应改为在循环体内显式关闭或封装成函数。

**1.8 ⭐⭐⭐ 逃逸分析是什么？如何判断变量分配在堆还是栈？**
- 编译期分析变量生命周期是否"逃出"函数栈帧。栈分配随函数返回自动回收（快、无 GC 负担）；逃逸到堆则由 GC 管理。
- 查看：`go build -gcflags="-m"`（会打印 escapes to heap）。
- 常见逃逸原因：返回局部变量指针、赋值给 interface（装箱）、闭包捕获引用、变量太大放不下栈、编译期大小不确定（如变长 slice）。
- 🧠 类比：栈像"随手用完就丢的草稿纸"（函数返回即撕掉），堆像"要归档、由清洁工（GC）定期清理的档案柜"。变量如果被外面还要用（返回指针/被闭包抓住），就只能进档案柜。
- 延伸：性能优化核心手段之一——减少逃逸就减少堆分配，进而降低 GC 频率；但不要为此写反直觉代码，先测量。

**1.9 ⭐⭐ 值接收者 vs 指针接收者，方法集有何区别？**
- 值接收者 `func (t T)`：`T` 和 `*T` 都能调用，方法内操作的是**副本**。
- 指针接收者 `func (t *T)`：能修改原对象；`*T` 的方法集包含指针方法，而 `T`（值）的方法集**不含**指针方法——影响接口实现判定。
- 🧠 类比：值接收者像"给你一张复印件"，你怎么改都不影响原件；指针接收者像"给你原件的地址"，你能改原件。想改数据 / 结构体大 → 用指针。
- 延伸：一个类型的方法要么统一用值、要么统一用指针（避免混用引发方法集混乱）；含 Mutex 的结构体必须用指针接收者（值拷贝会复制锁）。

**1.10 ⭐ context 的作用？如何正确传播取消与超时？**
- 作用：跨 API 和 goroutine 传递**取消信号、超时/截止时间、请求作用域的值**（trace id 等）。
- `WithCancel/WithTimeout/WithDeadline/WithValue`；返回的 cancel 必须调用（`defer cancel()`）否则泄漏；约定作为函数**第一个参数**。
- 🧠 类比：context 像"包工头手里的哨子 + 闹钟"——一吹哨（cancel）所有干活的工人（goroutine）都停手；闹钟响（timeout）自动吹哨。哨声会一层层传给所有下级（父 ctx 取消，子 ctx 全取消）。
- 延伸：`WithValue` 只放请求域元数据，别用来传函数参数；key 用自定义类型避免碰撞。

**1.11 ⭐⭐ new 和 make 的区别？（补充）**
- `new(T)` 分配零值内存返回 `*T`，适用于任意类型；`make` 只用于 slice/map/channel，返回初始化好的**引用类型本身**（非指针）。
- 🧠 类比：`new` 是"买个空盒子给你钥匙（指针）"；`make` 是"买来并组装好能直接用的家电（slice/map/chan）"。

**1.12 ⭐⭐ 字符串和 []byte 的区别与转换开销？（补充）**
- string 不可变（底层 `{ptr, len}`），`[]byte` 可变。互转默认会**拷贝内存**（保证不可变性）。
- 🧠 类比：string 是"刻在石碑上的字"（不可改），[]byte 是"白板上的字"（可擦写）。互转就是把内容抄一份。
- 延伸：高频转换用 `strings.Builder`、`unsafe`（谨慎）或复用 buffer 避免拷贝开销。

**1.13 ⭐⭐ init 函数和包初始化顺序？（补充）**
- 顺序：依赖包先初始化 → 包级变量按依赖顺序 → `init()`（一个文件可多个，按出现顺序；多文件按文件名字典序）。同一包 init 在 main 前执行。
- 🧠 类比：像"装修房子"——先通水电（依赖包）、再摆家具（变量）、最后做收尾检查（init），全部就绪才能入住（main）。

## 2. 并发编程：goroutine / channel / 同步原语

**2.1 ⭐ channel 有缓冲和无缓冲的区别？关闭已关闭 channel 会怎样？**
- 无缓冲：发送和接收必须**同时到场**才能完成（同步握手）；有缓冲：缓冲没满发送不阻塞，没空接收不阻塞。
- 向已关闭 channel 发送 → panic；关闭已关闭 channel → panic；从已关闭 channel 读 → 立即返回零值 + `ok=false`。
- 🧠 类比：无缓冲 channel 像"当面交接文件"，双方都得在场；有缓冲像"文件柜（容量 N）"，投件方只要柜子没满就能塞、取件方只要柜子没空就能拿。关闭 channel 像"贴上'已停用'封条"，再往里塞（发送）会出事，重复封（关闭）也会出事。
- 延伸：关闭原则「谁发送谁关闭，且只关一次」；多发送者场景用 `sync.Once` 或额外的信号 channel 协调关闭。

**2.2 ⭐⭐ select 的执行机制？如何实现超时和非阻塞收发？**
- 多个 case 同时就绪时**随机**选一个（防止饥饿/依赖顺序）；都没就绪且有 `default` 就走 default（非阻塞）。
- 超时：`select { case v := <-ch: ...; case <-time.After(t): // 超时 }`；非阻塞：加 `default`。
- 🧠 类比：select 像"多个窗口同时办业务，哪个先叫到号就去哪个"，都叫到了就随机挑一个；`default` 像"都没空就先干别的别干等着"。
- 延伸：`time.After` 在 for-select 循环里每轮都新建 timer 且到期前不回收，高频场景会内存泄漏；应改用 `time.NewTimer` + `Reset` 复用。

**2.3 ⭐⭐⭐ 如何优雅控制 goroutine 生命周期、避免泄漏？**
- 用 context 传取消；worker 用 `select { case <-ctx.Done(): return; case job := <-ch: ... }` 监听退出；确保每个 channel 都有出口不会永久阻塞。
- 常见泄漏：向没有接收者的 channel 发送、`range` 一个永不关闭的 channel、忘记调用 cancel、WaitGroup 计数不匹配。
- 排查：`pprof goroutine` 看堆栈聚集处、`runtime.NumGoroutine()` 监控趋势。
- 🧠 类比：goroutine 泄漏像"派出去的外卖员一直在门口等一个永远不开门的客户"，越积越多。给每个外卖员配"召回哨（ctx）"和"最长等待时间（超时）"就不会无限等。
- 延伸：一句口诀——**启动一个 goroutine 前，先想清楚它怎么退出**。

**2.4 ⭐⭐ sync.Mutex 与 RWMutex 的适用场景？Mutex 的正常/饥饿模式？**
- 读多写少 → RWMutex（多个读可并行）；写频繁或临界区极短 → 普通 Mutex（RWMutex 的读写记账反而更慢）。
- Mutex 两种模式：正常模式（新来的 goroutine 可能"插队"抢锁，吞吐高）；等待超过 1ms 切**饥饿模式**，锁直接按 FIFO 交给队首，防止尾部长期饿死。
- 🧠 类比：RWMutex 像"图书馆阅览室"——多人可同时看书（读），但有人要重新排版（写）时得清场独占。饥饿模式像"排队太久的人被安排优先办理"，防止总被插队的人饿死。
- 延伸：锁粒度要小（只锁必要的临界区）；避免锁内做 IO / RPC；用 `defer unlock` 防漏解锁但注意热点路径的性能。

**2.5 ⭐⭐ sync.WaitGroup 使用注意点？Add 为什么要在 goroutine 外？**
- `Add` 必须在启动 goroutine **之前**调用；若在 goroutine 内部 Add，主协程可能先跑到 `Wait` 发现计数为 0 提前返回。
- WaitGroup 不能被复制（内含 noCopy）；计数变负会 panic。
- 🧠 类比：WaitGroup 像"点名册"——出发前（启动 goroutine 前）先在册子上记下"共 N 人"（Add N），每人回来签退（Done），带队的等所有人签完（Wait）。如果边走边记名，可能还没记就以为人齐了。
- 延伸：常配合 errgroup 用于"并发子任务 + 收集错误"。

**2.6 ⭐⭐⭐ sync.Once 的实现原理？**
- 内部一个 `done` 原子标志 + 一把互斥锁。快路径先 atomic 读 done，已完成直接返回；慢路径加锁、双重检查、执行 f、置 done。
- 🧠 类比：像"景区检票口"——大多数人来了先瞄一眼"已开门"的牌子（atomic 读）直接进；第一个到的人负责开门（加锁执行），开完挂牌，后面的人就不用再开了。
- 延伸：为什么不能只用 atomic？纯 atomic 只能标记"开始了"，无法保证"执行**完成**"后其他人才通过，必须靠锁串行化并建立 happens-before。

**2.7 ⭐⭐ sync.Pool 的作用与陷阱？**
- 缓存可复用的临时对象，降低分配和 GC 压力；每个 P 有本地池，减少竞争。
- 陷阱：**每轮 GC 会清空 Pool**（不能当持久缓存/连接池用）；取出的对象状态是"上次用过的"，必须先 Reset。
- 🧠 类比：sync.Pool 像"公共雨伞架"——用完放回，下次别人可以拿现成的（省得买新的=省分配）；但清洁工（GC）来了会把伞架清空，所以别指望伞一直在。拿到的伞可能还带着上次的水（要先甩干=Reset）。
- 延伸：典型用于 bytes.Buffer、序列化临时缓冲、[]byte 缓冲。

**2.8 ⭐⭐⭐ 什么是数据竞争（data race）？如何检测和避免？**
- 定义：≥2 个 goroutine **并发**访问同一内存，且**至少一个是写**，且**没有同步**。结果不确定，可能读到半更新的值。
- 检测：`go test -race` / `go run -race`（运行期插桩，性能降但能抓现行）。避免：加锁 / channel 通信 / atomic / 只读共享（不可变）。
- 🧠 类比：data race 像"两个人同时在同一张白板上写字又互相看"——最后是谁的字、看到的是不是完整的字，全凭运气。加锁就是"约定一次只让一个人上手"。
- 延伸：口号——**不要通过共享内存来通信，要通过通信来共享内存**（channel 传递所有权）。data race ≠ race condition：前者是内存层并发访问，后者是逻辑时序依赖，范围更广。

**2.9 ⭐⭐ atomic 操作的适用场景？CAS 是什么？**
- 适用：计数器、标志位等**单个变量**的简单读改写，比锁轻量。CAS（CompareAndSwap）= "值还等于旧值才更新"，无锁乐观并发的基础。
- 🧠 类比：CAS 像"改共享文档前先确认'还是我上次看到的版本吗？'，是才提交，否则重试"——乐观锁思想。
- 延伸：atomic 只能保证**单个变量**原子，多个变量的复合操作（如同时改两个字段）仍需锁；`atomic.Value`/`atomic.Pointer` 可原子替换整个对象（配置热更新常用）。

**2.10 ⭐⭐⭐ 实现一个并发安全、限制并发数的任务池（题目）。**
- 思路一：带缓冲 channel 当信号量——`sem := make(chan struct{}, N)`，进任务前 `sem<-struct{}{}`，完成后 `<-sem`。
- 思路二：固定 N 个 worker + 一个任务 channel，worker `range` 任务 channel 消费。
- 考察点：优雅关闭（关任务 channel + WaitGroup 等待）、错误收集、结果聚合、context 取消联动、panic 恢复。
- 🧠 类比：信号量像"停车场 N 个车位"，来车先取一张停车卡（占位），没卡了就等；走了还卡（腾位）。worker 池像"N 个固定收银员轮流处理排队顾客"。

**2.11 ⭐⭐ errgroup 的作用？和 WaitGroup 有什么区别？**
- errgroup = WaitGroup + 首个错误返回 + context 联动取消（一个子任务出错，`ctx` 取消，其余尽快停）；`SetLimit(n)` 可限并发。
- 🧠 类比：WaitGroup 只负责"等所有人回来"；errgroup 还多了"谁先出事就通知全队撤退，并把出事原因带回来"。
- 延伸：适合"扇出多个依赖查询、任一失败即整体失败"的场景（如聚合多个下游）。

**2.12 ⭐⭐ Go 里如何做超时控制的几种方式？（补充）**
- context.WithTimeout（推荐，可级联传播）、`select + time.After`、`time.AfterFunc`。RPC/DB/HTTP 客户端都应设超时。
- 🧠 类比：没有超时的调用像"打电话没人接还一直握着听筒"，占着线路（连接/goroutine）不放，最终线路耗尽=雪崩。

**2.13 ⭐⭐ 什么是 happens-before？为什么并发要靠它？（补充，衔接内存模型）**
- 只有建立了 happens-before 关系，一个 goroutine 的写才**保证**对另一个可见。channel 收发、锁、Once、atomic、WaitGroup 都能建立这种关系。
- 🧠 类比：像"接力赛交棒"——只有棒（同步点）交到手上，你才确定上一棒跑完了、能看到他的成果；没交棒就各跑各的，看到什么全不保证。

## 3. 内存模型、GC 与性能调优

**3.1 ⭐⭐⭐ Go 的 GC 算法是什么？三色标记 + 写屏障如何工作？**
- 并发标记清除（非分代、非压缩）。三色抽象：白=未访问（待回收）、灰=已发现待扫描、黑=已扫描完毕。从根（栈、全局变量）出发，灰变黑、把它引用的白变灰，直到没有灰色，剩下的白色回收。
- 难点：标记和程序并发跑，程序可能"黑对象新指向一个白对象、又断开灰对象到它的引用"导致白对象被误回收。**混合写屏障**（Go 1.8+）在指针写操作时做记录，保证不漏标，把 STW 压到亚毫秒级。
- 🧠 类比：三色标记像"给房子里的东西贴标签大扫除"——从门口（根）开始，看得见的先贴灰，检查完贴黑，同时把它连着的东西也贴灰；最后还是白的（没人用的）就扔掉。写屏障像"打扫时有人偷偷搬东西，门口安个记录员，凡是搬动都登记，保证不误扔还在用的东西"。
- 延伸：STW 只发生在标记开始（开启写屏障）和标记结束（关闭写屏障）两个极短点，其余并发进行；Go GC 优化目标是**低延迟**而非最高吞吐，所以不做分代/压缩。

**3.2 ⭐⭐ GC 何时触发？GOGC 和 GOMEMLIMIT 的作用？**
- GOGC（默认 100）：当**堆增长到上次 GC 后存活对象的 2 倍**（100%）时触发。调大→GC 更少但内存占用高；调小→更省内存但 GC 更频繁。
- GOMEMLIMIT（1.19+）：软内存上限，接近上限时更激进 GC，防容器 OOM。
- 🧠 类比：GOGC 像"垃圾桶多满才倒"的阈值——设 100 就是"攒到上次清完后的 1 倍新垃圾就倒一次"。GOMEMLIMIT 像"房间总容量红线"，快到红线就疯狂清理别撑爆。
- 延伸：容器里务必设 GOMEMLIMIT ≈ 容器内存 limit 的 ~90%，并配合 GOMAXPROCS 适配 CPU limit（否则默认读宿主机核数，调度/GC 失衡）。

**3.3 ⭐⭐⭐ 线上内存持续上涨如何排查内存泄漏？**
- 用 `pprof heap`：`inuse_space` 看当前占用、`alloc_space` 看累计分配；抓多次快照对比增长点定位分配调用栈。
- 常见根因：goroutine 泄漏（每个 goroutine 持有对象）、全局 map/缓存只增不减、slice/substring 引用大底层数组、资源未 Close、`time.Ticker`/`Timer` 未 Stop、cgo 内存。
- 🧠 类比：内存泄漏像"水池只进不出"，先找哪个水龙头一直开着（哪段代码持续分配又不释放）。pprof 快照对比像"每隔一会拍张水位照片，看哪块区域水位一直涨"。
- 延伸：要区分三种"涨"——真泄漏（对象一直被引用）、GC 未及时（临时高峰）、RSS 未归还 OS（Go 会缓存内存不马上还，属正常）。

**3.4 ⭐⭐⭐ CPU 使用率高如何定位热点？**
- `pprof cpu` 采样 30s + 火焰图（`go tool pprof -http=:8080`），火焰图越宽的函数占 CPU 越多。
- 关注：频繁分配触发 GC（GC 占 CPU）、锁竞争（mutex/block profile）、序列化/正则/加解密热点、无谓的反射。
- 🧠 类比：火焰图像"从上往下看的一叠账单"，哪一层横条最长，钱（CPU）就花在那；顺着往下钻就能找到最烧钱的那笔。
- 延伸：`us` 高看应用代码、`sy` 高看系统调用（如频繁 write/网络）、`GC` 高看分配。

**3.5 ⭐⭐ 如何降低 GC 压力？**
- 减少堆分配：预分配 slice/map 的 cap、复用对象（sync.Pool）、小对象值传递、避免不必要的指针和 interface 装箱、减少逃逸。
- 🧠 类比：GC 压力大像"清洁工被海量小垃圾累垮"。少产生垃圾（少分配）、能重复用的碗别每次换新的（对象复用），清洁工就轻松了。
- 延伸：`GODEBUG=gctrace=1` 打印每轮 GC 的耗时、频率、堆大小，是判断"GC 是不是瓶颈"的第一手数据。

**3.6 ⭐⭐ Go 内存模型中的 happens-before 是什么？**
- 规定跨 goroutine 的内存操作可见性顺序。没有同步原语建立 happens-before，就不能假设一个 goroutine 的写对另一个可见（编译器/CPU 可能重排）。
- 建立方式：channel 收发、Mutex Lock/Unlock、Once.Do、atomic、WaitGroup、go 语句启动。
- 🧠 类比：见 2.13 接力交棒。核心记忆：**共享数据的可见性不是"写了就能看到"，而是"同步过才能看到"**。

**3.7 ⭐⭐ pprof 的几种 profile 类型分别用途？**
- cpu（热点函数）、heap（内存分配/占用）、goroutine（协程堆栈与数量）、block（阻塞在同步原语的时间）、mutex（锁竞争）、trace（调度/GC/网络的时间线全景）。
- 🧠 类比：pprof 像"体检套餐"——cpu 查心率（谁最累）、heap 查体重（内存胖在哪）、goroutine 查人数（是否人满为患）、mutex/block 查排队（哪堵）、trace 是"全天行为录像"。
- 延伸：生产环境用 `net/http/pprof` 暴露端点（注意鉴权，别裸奔）；trace 用 `runtime/trace` 采集后 `go tool trace` 看。

**3.8 ⭐⭐⭐ 一次延迟抖动（P99 突刺）如何定位？**
- 排查维度：GC STW / mark assist（分配太快被拉去帮忙标记）、锁竞争、goroutine 暴涨导致调度排队、下游依赖（DB/缓存/RPC）抖动、连接池耗尽、GC 触发点与流量高峰重叠。
- 工具：`runtime/trace` 看 STW 和调度延迟、metrics（GC pause、goroutine 数、P99）、分布式链路追踪定位是本服务还是下游耗时。
- 🧠 类比：P99 突刺像"高速公路平时畅通但偶尔堵一下"——可能是清障车临时封道（GC STW）、收费站排队（锁/连接池）、前方路段（下游）事故。要装"行车记录仪（trace/链路）"才能还原那一瞬间。
- 延伸：mark assist 是隐蔽元凶——分配速率过高时，分配的 goroutine 会被强制"打工帮 GC 标记"，表现为个别请求莫名变慢。降低分配速率或调 GOGC 缓解。

**3.9 ⭐⭐ 栈和堆的区别？Go 的 goroutine 栈是怎么增长的？（补充）**
- 栈：函数私有、随返回自动回收、访问快；堆：跨函数共享、GC 管理。goroutine 初始栈仅 2KB，不够时**分段/连续栈扩容**（拷贝到更大栈），可缩容。
- 🧠 类比：goroutine 栈像"可伸缩的抽屉"，初始很小（2KB，所以能开百万协程），东西多了自动换更大的抽屉并搬过去。这也是 Go 能轻量开海量协程的原因（对比线程默认几 MB 栈）。

**3.10 ⭐⭐ benchmark 怎么写？如何避免测试被编译器优化掉？（补充）**
- `func BenchmarkXxx(b *testing.B)` + `for i:=0;i<b.N;i++`；`b.ReportAllocs()` 看分配；结果赋给包级变量或用 `runtime.KeepAlive` 防被优化消除。
- 🧠 类比：benchmark 像"跑步机测心率"，`b.N` 是系统自动决定的"跑多少步才测得准"。

## 4. 微服务与 RPC

**4.1 ⭐⭐ gRPC 相比 REST 的优势？底层依赖什么？**
- 基于 HTTP/2（多路复用、头压缩、二进制分帧、双向流）+ Protobuf（紧凑二进制、强类型、代码生成）。四种模式：一元、服务端流、客户端流、双向流。
- 🧠 类比：REST/JSON 像"写信用大白话，人人能读但啰嗦"；gRPC/Protobuf 像"用双方约定的电报密码，短小高效但需先对暗号（.proto）"。HTTP/2 多路复用像"一条电话线同时并行多路通话，不用一件件排队"。
- 延伸：REST 适合对外开放 API、浏览器友好、调试直观；gRPC 适合内部服务间高性能通信。gRPC 也有 grpc-gateway 转 REST。

**4.2 ⭐⭐ Protobuf 如何保证向前/向后兼容？**
- 字段靠**编号（tag）**标识而非名字。规则：新增字段用**新编号**；不删除/不复用旧编号（用 `reserved` 保留）；不改已有字段类型；老服务遇到不认识的新字段会跳过。
- 🧠 类比：像"表格按列号而非列名对齐"——加新列放到末尾新列号，老程序看不懂新列就忽略，新程序遇到缺列用默认值，双方都不崩。
- 延伸：proto3 所有字段"可选"、标量默认零值，无法区分"没传"和"传了零值"（需 wrapper 或 optional 关键字解决）。

**4.3 ⭐⭐⭐ RPC 超时、重试、幂等如何设计？**
- 超时：用 context deadline **逐级传递**，下游剩余时间随链路递减。
- 重试：指数退避 + 抖动（jitter）+ 最大次数上限；只对"可安全重试"的错误（网络超时、连接失败、5xx 非业务错误）重试，业务错误（如余额不足）不重试。
- 幂等：重试的前提是接口幂等（唯一请求 id / 幂等 token / 去重表）。
- 🧠 类比：重试像"电话没打通就再拨"，但要有礼貌——间隔越来越长（退避）、别所有人同时重拨（抖动，避免"重试风暴"打垮下游）、拨几次不通就放弃（上限）。幂等像"同一笔转账重复提交，系统认得出是同一笔，只扣一次钱"。
- 延伸：重试放大效应——A→B→C 每层各重试 3 次，C 会收到 9 倍甚至 27 倍流量，只应在**最靠近失败的一层**重试或用重试预算（retry budget）。

**4.4 ⭐⭐ 服务注册与发现如何实现？**
- 注册中心（etcd/Consul/Nacos）。服务启动时注册地址 + 定期心跳续约（租约 lease），异常下线后租约过期自动摘除；调用方拉取实例列表 + watch 变更做客户端负载均衡。
- 🧠 类比：像"外卖平台的骑手在线状态"——骑手上线登记（注册），每隔几秒上报"我还在"（心跳），断线一会儿平台就标记离线（租约过期摘除），派单系统只从在线骑手里选（发现）。
- 延伸：健康检查（主动探测 vs 心跳上报）、优雅下线（先摘注册再停服务，避免流量打到已停实例）。

**4.5 ⭐⭐ 常见负载均衡策略？客户端 vs 服务端负载均衡？**
- 策略：轮询、加权轮询、随机、最少连接、一致性哈希（会话/缓存亲和）、P2C（随机选两个挑负载低的，兼顾均衡与低开销）。
- 客户端 LB：调用方自己选实例（gRPC、服务网格 sidecar），少一跳、延迟低；服务端 LB：经 Nginx/LB 集中转发，管理统一但多一跳。
- 🧠 类比：P2C 像"排队时随便看两个窗口，去人少的那个"——比全局找最优省事，又比纯随机均衡。一致性哈希像"同一个 VIP 客户总分给同一个专属客服"（缓存/会话命中率高）。

**4.6 ⭐⭐⭐ 分布式链路追踪如何落地？TraceID 如何透传？**
- OpenTelemetry / Jaeger / SkyWalking。一次请求生成全局 trace id，每个环节生成 span id 记录耗时和父子关系；通过 context + RPC metadata（HTTP header / gRPC metadata）透传。
- 🧠 类比：trace 像"快递全程物流追踪号"——一个总单号（trace id）串起每个中转站（span），每站记录几点到、几点走、上一站是谁，最后拼出完整时间线看哪段最慢。
- 延伸：采样策略（头部采样省成本 vs 尾部采样只保留出错/慢的更精准）；跨异步边界（MQ）要把 trace 信息塞进消息头才能续接。

**4.7 ⭐⭐ 服务间通信如何保证安全？**
- mTLS 双向证书认证、服务身份（SPIFFE）、token/JWT 鉴权、服务网格统一策略、网络分段隔离。
- 🧠 类比：mTLS 像"双方都要出示工作证"——不只是我确认你（单向 TLS），你也要确认我，冒充者进不来。
- 延伸：内网也不应"默认信任"（零信任理念），东西向流量同样鉴权。

**4.8 ⭐⭐⭐ 微服务拆分的原则？拆得过细有什么问题？**
- 按业务领域边界（DDD 限界上下文）、变更频率、团队边界拆；追求高内聚低耦合，一个服务拥有自己的数据（不共享库）。
- 过细的代价：分布式复杂度飙升（原本函数调用变网络调用）、跨服务事务困难、链路变长排障难、运维/部署成本高、性能损耗。
- 🧠 类比：拆服务像"分部门"——按职能分（销售/财务）而不是随便切。分太细就像"每个人一个部门"，开会（网络调用）成本、扯皮（一致性）成本全上来了。
- 延伸：不确定边界时先"模块化单体"，边界清晰后再拆；避免"分布式单体"（拆了但强耦合、必须一起发布）。

**4.9 ⭐⭐ RPC 序列化协议如何选型？（补充）**
- Protobuf/Thrift（二进制、快、需 schema）、JSON（可读、通用、慢）、MessagePack、Avro（大数据）。内部服务选二进制，对外/调试选 JSON。
- 🧠 类比：见 4.1 电报密码 vs 大白话。

**4.10 ⭐⭐ 什么是 API 网关？它承担哪些职责？（补充）**
- 统一入口：路由转发、鉴权认证、限流熔断、协议转换、日志监控、灰度分流。
- 🧠 类比：API 网关像"公司前台 + 保安"——所有访客统一从这进，验身份、登记、指路、拦截可疑人员，内部各部门（微服务）不用各自设岗。

## 5. 消息队列（Kafka / RabbitMQ）

**5.1 ⭐⭐ Kafka 为什么高吞吐？**
- 顺序写磁盘（比随机写快几个数量级）、利用 OS page cache、零拷贝（sendfile 省去内核↔用户态拷贝）、批量发送 + 压缩、分区并行、消费者 pull 模式自主控速。
- 🧠 类比：顺序写像"往笔记本一页页往后写"（快），随机写像"每次翻到不同页写一行"（慢，磁盘磁头乱跑）。零拷贝像"仓库直接发货给客户，不先搬到办公室再转手"。
- 延伸：Kafka 把"日志顺序追加"这一磁盘最擅长的模式用到极致，这是它高吞吐的根本。

**5.2 ⭐⭐⭐ Kafka 如何保证消息不丢失？（三端配合）**
- 生产端：`acks=all`（等所有 ISR 副本确认）+ `min.insync.replicas>=2` + 开启重试。
- Broker 端：多副本 + 合理刷盘/复制策略，leader 挂了从 ISR 选新 leader。
- 消费端：**先处理业务，再手动提交 offset**（关闭自动提交），避免"提交了但没处理完就崩"。
- 🧠 类比：像"寄重要包裹"——寄件时要求"所有分拣中心都签收才算发出"（acks=all），中转站多点备份（多副本），收件人"确认收到并拆封验货后才签字"（先处理后提交 offset）。任一环节偷懒都可能丢件。
- 延伸：`acks=1`（只等 leader）在 leader 切换瞬间可能丢；`acks=0` 最快但最不可靠。

**5.3 ⭐⭐⭐ 如何保证消息不重复消费（幂等 / Exactly-Once）？**
- 消费幂等：唯一业务 id + 去重表 / `Redis SETNX` / 数据库唯一键，重复消息直接丢弃。
- 生产幂等：`enable.idempotence=true`（PID + 单调序列号，Broker 去重）。
- Exactly-Once：Kafka 事务（生产 + offset 提交在一个事务里，配合读已提交隔离）。
- 🧠 类比：幂等去重像"演唱会门票核销"——同一张票（业务 id）第一次扫进场，第二次扫显示"已使用"直接拦下，无论你扫几次结果一样。
- 延伸：分布式下"恰好一次"投递极难，工程上普遍做法是"**至少一次投递 + 消费端幂等**"= 效果上的恰好一次。

**5.4 ⭐⭐ Kafka 如何保证消息顺序？**
- 单分区内消息有序；把需要保序的消息用**同一个 key** 路由到同一分区（如同一订单的所有事件）。全局有序需单分区（牺牲并行度）。
- 🧠 类比：分区像"多条结账通道"，每条通道内先来后到（有序），但通道之间不保证谁先结完。想让同一顾客的商品按顺序过，就固定让他走同一条通道（相同 key）。
- 延伸：消费端若用多线程处理会打乱顺序——需按 key 哈希到固定线程处理，或单线程消费。

**5.5 ⭐⭐ 消费者组和 rebalance 机制？rebalance 的问题？**
- 同一消费组内，一个分区只被一个消费者消费（实现负载均分）；成员增减/分区变化触发 rebalance 重新分配。
- 问题：rebalance 期间**全组暂停消费**（stop-the-world）；优化——静态成员（group.instance.id）、增量协作式 rebalance（cooperative sticky）、合理设置 session.timeout / heartbeat。
- 🧠 类比：消费组像"几个搬运工分包裹传送带"，每条带子（分区）只由一人负责。有人来了或走了就要重新分配带子（rebalance），分配期间大家先停手，所以频繁进出很影响效率。
- 延伸：消费太慢导致心跳超时被踢 → 触发 rebalance → 更慢，形成恶性循环；`max.poll.interval.ms` 要大于单批处理耗时。

**5.6 ⭐⭐ 消息积压如何处理？**
- 先定位原因：消费慢 / 生产突增 / 下游故障。应急：临时扩分区 + 扩消费者实例、提升消费并发、批量处理；极端情况旁路——快速把消息转存到别处后续慢慢处理，先恢复生产。
- 🧠 类比：积压像"传送带上包裹堆成山"——加人手（扩消费者，但不能超过分区数）、加通道（扩分区）、或先把包裹搬到临时仓库（旁路转存）别堵住入口。
- 延伸：消费者数量 > 分区数时多余的消费者空闲，所以扩消费者前先扩分区。

**5.7 ⭐⭐ Kafka vs RabbitMQ 如何选型？**
- Kafka：超高吞吐、消息可回溯（按 offset 重放）、日志/流处理/大数据管道、分区水平扩展。
- RabbitMQ：低延迟、灵活路由（exchange 的 direct/topic/fanout）、复杂投递语义、传统业务解耦、消息优先级/延迟插件。
- 🧠 类比：Kafka 像"大流量的传送带/数据河流"，适合海量日志顺流而下且可回放；RabbitMQ 像"灵活的邮局分拣系统"，规则多、投递花样多、响应快。
- 延伸：RocketMQ 折中——高吞吐 + 事务消息 + 延迟消息，国内互联网常用。

**5.8 ⭐⭐⭐ 如何用 MQ 实现最终一致性？**
- 本地消息表：业务数据和"待发消息"写进**同一个本地事务**，后台任务轮询消息表投递到 MQ，投递成功标记完成。保证"业务成功则消息一定发出"。
- 事务消息（RocketMQ/Kafka）：先发"半消息"（对消费者不可见）→ 执行本地事务 → 提交/回滚半消息；长时间未确认则 Broker 回查本地事务状态。
- 🧠 类比：本地消息表像"把'要寄的信'和'办完的事'记在同一个账本里一起落笔"——账本记上了，就算此刻断电，恢复后照着账本一定把信寄出去，不会出现"事办了信没寄"。
- 延伸：这是解决"数据库操作 + 发消息"两个动作无法原子的经典方案（不能直接用分布式事务时的首选）。

**5.9 ⭐⭐ 死信队列（DLQ）和延迟消息的用途？**
- DLQ：消息多次消费失败后转入死信队列隔离，避免毒消息（poison message）反复失败堵塞正常消费，便于人工排查/补偿。
- 延迟消息：定时执行场景——超时未支付关单、N 分钟后发提醒。
- 🧠 类比：DLQ 像"快递里的'疑难件仓库'"——反复投不成功的件先挪到这，别堵住正常派送线，再单独处理。延迟消息像"定时闹钟提醒"。

**5.10 ⭐⭐ 推模式 vs 拉模式的区别？Kafka 为什么用 pull？（补充）**
- 推（push）：Broker 主动推，实时但可能压垮慢消费者；拉（pull）：消费者按自己节奏取，可批量、可控速、天然背压。
- 🧠 类比：push 像"服务员不停给你上菜，吃不完也堆桌上"；pull 像"自助餐自己去取，吃多少拿多少"，不会被撑到。

## 6. 缓存与 Redis

**6.1 ⭐ Redis 为什么快？单线程为什么不慢？**
- 纯内存操作、高效数据结构、IO 多路复用（epoll 单线程管海量连接）、单线程避免了锁竞争和线程上下文切换。6.0+ 网络 IO 用多线程，但**命令执行仍单线程**保证原子性。
- 🧠 类比：Redis 像"一个超快的收银员用 epoll 同时盯着几百个自助结账口"——因为每单都极快（内存操作），一个人反而比多人抢一个收银机（加锁）更高效。
- 延伸：单线程也意味着**一个慢命令（如 `KEYS *`、大 key 操作）会阻塞所有请求**，生产禁用这类命令。

**6.2 ⭐⭐ 常见数据结构及应用场景？**
- String（缓存/计数/分布式锁）、Hash（存对象字段）、List（消息队列/最新列表）、Set（去重/共同好友交集）、ZSet（排行榜/延迟队列，按 score 排序）、Bitmap（签到/在线状态）、HyperLogLog（海量 UV 基数估算，省内存有误差）、Geo（附近的人）。
- 🧠 类比：ZSet 像"带分数的排行榜"，天然排序还能按分数范围查；HyperLogLog 像"用一小把沙子估算体育场人数"——不精确但极省空间。
- 延伸：Stream（5.0+）是 Redis 原生消息队列，支持消费组。

**6.3 ⭐⭐⭐ 缓存穿透、击穿、雪崩的区别与解决方案？**
- **穿透**：查一个数据库里根本不存在的 key，缓存永远不命中，请求全打到 DB。解决：布隆过滤器提前拦截、缓存空值（短 TTL）。
- **击穿**：某个**热点 key** 恰好过期的瞬间，大量并发同时打到 DB。解决：互斥锁只让一个请求重建缓存、逻辑过期（不真过期，后台异步刷新）、热点 key 永不过期。
- **雪崩**：大量 key **同一时间**集中过期，或 Redis 整体宕机，流量瞬间压垮 DB。解决：过期时间加随机抖动、多级缓存、限流降级、Redis 集群高可用。
- 🧠 类比：穿透="查一个不存在的房间号，保安每次都得跑去总台确认"（布隆过滤器=门口先有张'有效房号名单'）；击穿="某明星房间的通行证刚好过期，粉丝全涌向前台"（加锁=只放一个人去补办）；雪崩="整栋楼的门禁同时失效"（错开过期时间=让门禁分批过期）。
- 延伸：三者记忆口诀——**穿透查无（不存在）、击穿单点（热点过期）、雪崩批量（集中失效）**。

**6.4 ⭐⭐⭐ 缓存与数据库一致性如何保证？**
- 主流 Cache Aside：读时缓存没有则查 DB 回填；写时**先更新 DB，再删除缓存**（不是更新缓存）。
- 为什么删而不是改：避免并发写导致缓存脏值、避免写了缓存却没人读白算（懒加载）。
- 延迟双删：写后删一次，隔一小段再删一次，兜住"删缓存后、DB 提交前有请求读了旧值回填"的并发窗口。
- 强一致方案：订阅 binlog（canal）异步更新缓存、分布式锁串行化。
- 🧠 类比：缓存像"贴在门口的今日菜单副本"，后厨（DB）改了菜单就把门口副本撕掉（删缓存），下一个客人来看时发现没副本就去后厨抄一份最新的（回填）。延迟双删=撕完过一会再撕一次，防止有人在你撕完的空档又贴了张旧的。
- 延伸：缓存一致性只能做到"最终一致"，追求强一致代价极高；先问业务能否容忍短暂不一致。

**6.5 ⭐⭐ Redis 持久化 RDB 和 AOF 的区别？如何选择？**
- RDB：定时快照，文件小、恢复快，但两次快照间宕机会丢数据。AOF：记录每条写命令，丢失少（可每秒 fsync），但文件大、恢复慢。
- 🧠 类比：RDB 像"定期给内存拍全景照片"，照片之间发生的事就丢了；AOF 像"记流水账"，每笔都记，恢复时重放账本。
- 延伸：生产常用**混合持久化**（RDB 全量 + AOF 增量），兼顾恢复速度和数据安全。

**6.6 ⭐⭐⭐ Redis 集群方案对比？如何扩容？**
- 主从复制（读写分离基础，不高可用）；哨兵 Sentinel（监控 + 自动故障转移，主挂了选新主）；Cluster（数据分片到 16384 个槽，多主横向扩展）；Proxy 分片（Codis/twemproxy）。
- Cluster 扩容：把部分哈希槽迁移到新节点，客户端访问迁移中的 key 会收到 MOVED/ASK 重定向。
- 🧠 类比：主从像"主账本 + 备份抄本"；哨兵像"监工，主账本管理员倒下就立刻指定备份管理员顶上"；Cluster 像"把 16384 个储物格分给多个管理员各管一段"，人手不够就把一部分格子交接给新管理员（迁移槽）。
- 延伸：Cluster 下 `MULTI` 事务/Lua 要求 key 在同一槽（用 hash tag `{}` 强制同槽）。

**6.7 ⭐⭐ 如何实现分布式锁？Redis 分布式锁的坑？**
- `SET key uniqueVal NX PX ttl` 加锁（NX 保证只有一个人拿到，PX 设过期防死锁），value 用唯一值（如 UUID），释放用 **Lua 脚本**"先比对 value 再删"防误删别人的锁。
- 坑：业务没执行完锁就过期 → 用 watchdog 自动续期（Redisson 思路）；主从切换锁丢失 → RedLock（多数派，有争议）。
- 🧠 类比：分布式锁像"公共会议室的门禁牌"——先到先拿（NX），牌子会自动到期免得有人忘还（PX 防死锁），还牌时先看"是不是我的牌"再还（Lua 防误还别人的）。事没办完牌快到期就续时间（watchdog）。
- 延伸：强一致场景（如金融扣款）别只依赖 Redis 锁，应结合 DB 唯一约束/乐观锁兜底。

**6.8 ⭐⭐ 大 key、热 key 问题如何治理？**
- 大 key（单 key 存了巨量数据）：拆分成多个小 key、用 `UNLINK` 异步删除（避免 `DEL` 阻塞）、避免大集合一次性操作。
- 热 key（单 key 访问量极高，打爆单节点）：本地缓存（多级缓存）、key 加随机后缀分散到多节点、读副本分流。
- 🧠 类比：大 key 像"一个塞满东西的巨型抽屉"，一开就卡住整个柜子（阻塞单线程）；热 key 像"所有人都挤同一个窗口"，得多开几个副本窗口分流。
- 延伸：大 key 是 Redis 单线程模型下延迟抖动的常见元凶，删除/序列化都可能阻塞。

**6.9 ⭐⭐ Redis 内存淘汰策略有哪些？**
- noeviction（默认，满了写报错）、allkeys-lru / volatile-lru（淘汰最久未用）、allkeys-lfu / volatile-lfu（淘汰最少使用，热点集中场景更优）、volatile-ttl（淘汰最快过期的）、random。
- 🧠 类比：LRU 像"整理房间时先扔最久没碰的东西"；LFU 像"先扔用得最少的东西"——如果有些东西偶尔用一次但很久没动，LRU 会误扔，LFU 更懂"高频=重要"。
- 延伸：allkeys-* 对所有 key 生效，volatile-* 只淘汰设了过期时间的 key。

**6.10 ⭐⭐ Redis 管道（pipeline）和 Lua 脚本的作用？**
- pipeline：一次性发多条命令减少网络往返（RTT），但不保证原子。Lua：多条命令在服务端原子执行（单线程期间不被打断）。
- 🧠 类比：pipeline 像"一次把购物清单全给店员，别一样一样跑腿"（省往返）；Lua 像"这几步必须一气呵成中间不许插队"（原子）。
- 延伸：Lua 里别写耗时逻辑（会阻塞整个 Redis）。

## 7. MySQL、索引与分库分表

**7.1 ⭐⭐ InnoDB 为什么用 B+ 树而不是 B 树 / 哈希 / 红黑树？**
- B+ 树"矮胖"：非叶子节点只存 key 不存数据，一个节点能放更多 key（扇出大），树高低（一般 3~4 层就能存千万级），查一次只需几次磁盘 IO。叶子节点用**双向链表**串联，范围查询、排序极友好。
- 哈希：只支持等值查询，不支持范围/排序；红黑树：二叉树太高，磁盘 IO 次数多；B 树：数据分散在每层，范围查询要跨层，且单节点存数据导致扇出小、树更高。
- 🧠 类比：B+ 树像"图书馆的分级索引卡"——目录页（非叶子）只写"哪类书在哪个区"（只存 key，一页能写很多），真正的书都在最底层书架（叶子）且按顺序排成一排（链表），找连续一批书（范围查询）顺着书架走就行。
- 延伸：磁盘 IO 是瓶颈，B+ 树的设计核心就是"用尽量少的 IO 定位数据"。

**7.2 ⭐⭐⭐ 聚簇索引和二级索引的区别？回表是什么？**
- 聚簇索引（主键索引）：叶子节点直接存**整行数据**，数据按主键物理有序。二级索引（普通索引）：叶子节点只存**主键值**。
- 回表：用二级索引查到主键后，还得拿主键去聚簇索引再查一次取完整行数据——两次查找。**覆盖索引**（要查的字段都在索引里）可避免回表。
- 🧠 类比：聚簇索引像"按身份证号排列的完整档案柜，找到号就拿到全部资料"；二级索引像"按姓名排的小卡片，卡片上只写了身份证号"，想看完整资料还得拿号去大档案柜再翻一次（回表）。如果你只想知道身份证号，看小卡片就够了（覆盖索引，不回表）。
- 延伸：主键不宜过长（二级索引都存主键，主键越长二级索引越占空间）；建议用自增主键（顺序插入，避免页分裂）。

**7.3 ⭐⭐⭐ 最左前缀原则？如何设计联合索引？**
- 联合索引 `(a,b,c)` 相当于建了 `(a)`、`(a,b)`、`(a,b,c)` 三个前缀索引；查询必须从最左列开始连续匹配才能用上索引，跳过 a 直接查 b 用不上。
- 设计：等值查询列放前、范围查询列放后（范围列之后的列无法用索引）、高区分度列靠前、尽量覆盖查询字段。
- 🧠 类比：联合索引像"电话簿按'姓→名→中间名'排序"——你知道姓能快速定位，知道姓+名更精确；但只知道名不知道姓，电话簿就帮不上忙（不满足最左前缀）。
- 延伸：`WHERE a=1 AND b>5 AND c=3`，索引 `(a,b,c)` 只能用到 a 和 b（b 是范围，c 用不上）；调成 `(a,c,b)` 可让 c 也生效。

**7.4 ⭐⭐⭐ 如何分析和优化慢查询？**
- 开慢查询日志定位 SQL → `EXPLAIN` 分析执行计划，重点看：`type`（连接类型，全表 ALL 最差、const/ref/range 较好）、`key`（实际用的索引）、`rows`（预估扫描行数）、`Extra`（`Using filesort` 额外排序、`Using temporary` 临时表、`Using index` 覆盖索引好信号）。
- 优化：加合适索引、避免索引失效、改写 SQL、分页优化、必要时冗余/反范式。
- 🧠 类比：`EXPLAIN` 像"导航出发前预览路线"——告诉你走哪条路（索引）、要经过多少路口（rows）、有没有堵车路段（filesort/临时表）。type=ALL 就是"没走高速全程走小路"。
- 延伸：`rows` 大 + `type=ALL` = 全表扫描重点优化对象；`Using filesort` 不一定是文件排序，指"没用索引完成排序"。

**7.5 ⭐⭐ 哪些情况会导致索引失效？**
- 索引列上用函数/运算（`WHERE YEAR(t)=2024`）、隐式类型转换（字符串列传数字）、`LIKE '%x'` 前置通配、`OR` 连接非索引列、不满足最左前缀、范围查询后的列、`!=`/`NOT IN`/`IS NOT NULL` 部分场景。
- 🧠 类比：索引像"字典的偏旁部首检字表"——你按部首查很快；但如果你把字先"变形"了（加函数），或只知道字的后半部分（`%x`），检字表就用不上，只能一页页翻（全表扫描）。
- 延伸：`WHERE YEAR(create_time)=2024` 改成 `WHERE create_time >= '2024-01-01' AND < '2025-01-01'` 就能用索引。

**7.6 ⭐⭐⭐ MySQL 事务隔离级别？MVCC 和幻读如何解决？**
- 四级：读未提交（脏读）、读已提交（RC，防脏读）、可重复读（RR，默认，防不可重复读）、串行化（全防但慢）。
- MVCC：通过 undo log 版本链 + ReadView（一致性视图）实现"快照读"——不加锁读到某个时间点的版本。
- 幻读：RR 下用**间隙锁（gap lock）+ 记录锁 = next-key lock** 锁住范围，防止别人在区间插入新行。
- 🧠 类比：MVCC 像"给数据拍快照"——你的事务开始时拍了张照，之后一直看这张照片（快照读），别人怎么改都不影响你看到的。间隙锁像"不仅锁住已有座位，连座位之间的空隙也占住"，防止有人在你查询的范围里塞新座位（幻读）。
- 延伸：快照读（普通 SELECT）靠 MVCC，当前读（`SELECT ... FOR UPDATE`/`LOCK IN SHARE MODE`/更新）靠加锁。

**7.7 ⭐⭐⭐ redo log、undo log、binlog 的区别？两阶段提交？**
- redo log：InnoDB 物理日志，记录"某页做了什么修改"，用于崩溃恢复（持久性），循环写。
- undo log：记录数据旧版本，用于事务回滚 + MVCC 快照读。
- binlog：Server 层逻辑日志，记录 SQL/行变更，用于主从复制、数据恢复，追加写。
- 两阶段提交：redo prepare → 写 binlog → redo commit，保证崩溃后 redo 和 binlog 状态一致（不会出现"数据改了但主从不一致"）。
- 🧠 类比：redo 像"施工时的'已完成动作'便签"，断电后照便签把没落实的补上（恢复）；undo 像"每步操作前拍的'改之前长啥样'照片"，要撤销就照片还原；binlog 像"官方施工记录"给分部门（从库）照着复刻。两阶段提交像"先预备签字、等官方记录也写好、再正式签字"，确保两本账一致。
- 延伸：redo 保证 crash-safe，binlog 保证复制，两者用 2PC 绑定一致，是 MySQL 可靠性的核心。

**7.8 ⭐⭐⭐ 分库分表的方案与拆分键选择？**
- 垂直拆分：按业务把不同表拆到不同库（用户库/订单库）、把大表的冷字段拆出去。水平拆分：同一张表按 range（时间/id 段）或 hash（取模/一致性哈希）把行分散到多张表/多个库。
- 拆分键（sharding key）选择：高频查询维度（如订单按 user_id）、分布均匀避免热点、尽量让常见查询能路由到单片。
- 🧠 类比：垂直拆分像"把杂货铺按品类分成多家专卖店"；水平拆分像"一家店客人太多，按'姓氏首字母'开多家分店分流"。拆分键选不好（如按时间且都查最新）会导致新店挤爆、老店闲置（热点）。
- 延伸：中间件 ShardingSphere / MyCat / 自研；能不拆就不拆（先读写分离、加缓存、归档冷数据），单表通常千万级、超大才考虑分表。

**7.9 ⭐⭐⭐ 分库分表后的难题：分页、跨库 join、全局 id、扩容？**
- 分页：`ORDER BY` 全局排序需从每个分片取 `offset+limit` 再归并，深分页代价大（优化：禁止深翻 / 二次查询法 / 用游标）。
- 跨库 join：避免，改为多次查询 + 应用层拼装，或冗余字段/宽表。
- 全局 id：雪花算法（趋势递增）、号段模式（DB 批量发号）、Redis incr。
- 扩容：一致性哈希 + 虚拟节点减少迁移量、双写迁移、按 2 的倍数扩容降低 rehash 范围。
- 🧠 类比：分库分表后像"账本被拆到多个分店"——查排行榜（分页排序）得把每家分店的前几名都拿来再汇总排（归并）；想把两类数据关联（join）就得跑几家店手动对（应用层拼装）；发会员号（全局 id）不能各店各发免得撞号，得统一发号机。
- 延伸：分页深翻是最常被追问的痛点，标准答法是"业务上禁止深翻 + 引导用游标/搜索"。

**7.10 ⭐⭐ 主从复制原理与主从延迟如何解决？**
- 原理：主库写 binlog → dump 线程发给从库 → 从库 IO 线程写 relay log → SQL 线程重放。有半同步、并行复制（多线程重放）优化。
- 延迟处理：对一致性要求高的读走主库、缓存标记刚写过的数据、并行复制、监控 `Seconds_Behind_Master`。
- 🧠 类比：主从像"老师在黑板上写（主库），学生抄笔记（从库）"——学生抄得慢就有延迟。刚讲的重点（刚写的数据）想立刻确认就直接问老师（读主库），别看还没抄完的笔记。
- 延伸："写后立即读"读到旧数据是主从延迟的典型坑，解决靠"读主 / 强制路由 / 缓存"。

**7.11 ⭐⭐ 深分页 `LIMIT 1000000,20` 为什么慢？如何优化？**
- MySQL 要先扫描并**丢弃**前 100 万行才返回 20 行，越翻越慢。优化：`WHERE id > last_max_id ORDER BY id LIMIT 20`（游标/记住上次位置）、延迟关联（先用覆盖索引查出主键再回表取数据）。
- 🧠 类比：深分页像"看一本书要翻到第 5 万页，你从第一页一页页数过去（扫描丢弃）"，太蠢。游标法像"用书签记住上次看到哪，下次直接从书签翻起"。
- 延伸：这题几乎必考，记住两个关键词——**游标（记住上次 id）+ 延迟关联（先查主键）**。

**7.12 ⭐⭐ 什么是范式与反范式？何时反范式？（补充）**
- 范式减少冗余（数据一致、省空间），反范式增加冗余（用空间/一致性换查询性能，减少 join）。高并发读场景常适度反范式。
- 🧠 类比：范式像"信息只存一处，改一处即可，但用时要到处拼"；反范式像"常用信息冗余抄几份，读起来快，但改的时候要同步多份"。

**7.13 ⭐⭐ 乐观锁和悲观锁的区别？（补充）**
- 悲观锁：先加锁再操作（`SELECT ... FOR UPDATE`），假设一定有冲突；乐观锁：不加锁，更新时用版本号/CAS 校验（`UPDATE ... WHERE version=?`），假设很少冲突，冲突了重试。
- 🧠 类比：悲观锁像"进厕所先反锁门"（假设有人抢）；乐观锁像"不锁门，出来时确认没人趁机进过（版本没变）才算数，被打断就重来"。读多写少用乐观锁，写多冲突频繁用悲观锁。

## 8. 分布式事务与一致性

**8.1 ⭐⭐ CAP 和 BASE 理论？**
- CAP：一致性(C)、可用性(A)、分区容错(P) 三者不可兼得。分布式系统网络分区（P）必然存在，所以实际是在**分区发生时**在 C 和 A 之间取舍（CP 还是 AP）。
- BASE：Basically Available（基本可用）、Soft state（软状态/中间态）、Eventually consistent（最终一致）——是对强一致的工程妥协。
- 🧠 类比：CAP 像"异地两个仓库网断了"——要么坚持"两边数据必须一致，宁可暂停服务"（CP），要么"先各自照常营业，等网通了再对账"（AP）。BASE 就是"允许暂时对不上账，但最终一定对上"。
- 延伸：不是"永远三选二"，平时网络正常 C 和 A 都能满足，CAP 的取舍只在分区那一刻发生。

**8.2 ⭐⭐⭐ 常见分布式事务方案及适用场景？**
- 2PC/3PC（XA）：强一致，协调者统一提交/回滚，但同步阻塞、协调者单点、性能差。
- TCC（Try-Confirm-Cancel）：业务层两阶段，Try 预留资源、Confirm 确认、Cancel 补偿；性能好但业务侵入强、需处理幂等/空回滚/悬挂。
- Saga：长事务拆成一串本地事务，每步配一个补偿操作，出错反向补偿；最终一致，适合流程长、跨多服务。
- 本地消息表 / 事务消息：异步最终一致（见 5.8）。
- 🧠 类比：2PC 像"婚礼司仪喊'大家都准备好了吗？'全答好才继续，中间谁没准备好全体等着"（阻塞）；TCC 像"订酒店先'预留房间'（Try），确认了才'正式入住'（Confirm），取消就'释放房间'（Cancel）"；Saga 像"多步旅行，某步出问题就一步步往回退票退房（补偿）"。
- 延伸：选型口诀——**能最终一致就别上强一致**；强一致(2PC/TCC) 代价高，多数互联网业务用 Saga/消息表做最终一致。

**8.3 ⭐⭐⭐ TCC 的空回滚、悬挂、幂等问题如何处理？**
- 空回滚：Try 还没执行（如网络问题没到），Cancel 先到了 → 记录事务状态，发现没 Try 过就直接返回成功，不做实际回滚。
- 悬挂：Cancel 已执行完，迟到的 Try 才到 → 检测到该事务已 Cancel，就拒绝执行 Try（否则预留的资源没人释放）。
- 幂等：Confirm/Cancel 可能重复调用 → 用全局事务 id + 状态表去重。
- 🧠 类比：空回滚像"还没下单就收到退款请求，系统得认出'这单根本没下过'，别真去退不存在的款"；悬挂像"退款都处理完了，下单请求才姗姗来迟，得拦住它别再占资源"。
- 延伸：这三个问题是 TCC 落地的核心难点，靠"事务状态表 + 全局事务 id"统一解决。

**8.4 ⭐⭐ 幂等设计的通用手段？**
- 数据库唯一约束、去重表/去重 key、状态机流转（只允许合法状态转移）、token 令牌（先领 token 再提交，用过即失效）、乐观锁版本号。
- 🧠 类比：幂等像"防止重复提交订单"——发一张一次性令牌（token），提交时核销，重复提交时令牌已失效就拦下。
- 延伸：幂等是分布式重试/消息去重的基石，几乎所有"可能重试"的接口都要设计幂等。

**8.5 ⭐⭐ 分布式 ID 生成方案对比？**
- UUID：本地生成无依赖，但无序（做主键导致 B+ 树频繁页分裂、索引大）。雪花算法（Snowflake）：时间戳 + 机器 id + 序列号，趋势递增、性能高，但依赖时钟。号段模式：从 DB 批量取一段 id 缓存本地发放，减少 DB 压力。Redis incr：简单但依赖 Redis。
- 雪花时钟回拨问题：机器时间被回拨会生成重复 id → 检测到回拨则等待/拒绝/借用备用位。
- 🧠 类比：UUID 像"随机长串车牌"，唯一但杂乱不好排序；雪花 id 像"按时间+窗口编号的排队号"，基本递增又不重复；号段模式像"售票窗口一次领 1000 张号，发完再领"，省得每张都去总台要。
- 延伸：做数据库主键优先选趋势递增的（雪花/号段），避免 UUID 导致的写入性能问题。

## 9. 高可用：限流、熔断、降级

**9.1 ⭐⭐⭐ 常见限流算法及区别？**
- 固定窗口计数器：简单，但窗口临界点可能双倍突刺（前后两窗各满会瞬间超）。滑动窗口：把窗口细分平滑掉临界问题。漏桶：请求先入桶，以**恒定速率**流出，强行整流削峰（但突发也被削）。令牌桶：以恒定速率放令牌，请求拿到令牌才通过，桶里攒着令牌**允许一定突发**。
- Go 实现：`golang.org/x/time/rate`（令牌桶）；分布式限流用 Redis + Lua 原子计数。
- 🧠 类比：漏桶像"漏斗"——不管你倒多快，下面匀速滴出（平滑但不允许突发）；令牌桶像"停车场每分钟发 N 张停车券，攒着的券可以让一波车集中进"（允许突发）。固定窗口的临界突刺像"12:59 和 13:00 各放 100 人，这 1 秒内挤进 200 人"。
- 延伸：想平滑用漏桶，想允许突发用令牌桶（互联网更常用令牌桶）。

**9.2 ⭐⭐ 熔断器的三种状态？**
- 关闭（正常放行，统计失败率）→ 失败率超阈值 → 打开（直接快速失败，不再打下游，给它喘息）→ 过一段时间 → 半开（放少量试探请求，成功则恢复关闭，失败则重新打开）。
- 🧠 类比：熔断器就是"家里的保险丝/空气开关"——电流过载（下游频繁失败）就跳闸（打开）保护电路，过会儿试着合闸（半开）看看是否恢复正常。
- 延伸：熔断是为了防止"雪崩"——下游挂了还一直打它，会拖垮自己（线程/连接耗尽）并向上传导。参考 Hystrix / Sentinel / gobreaker。

**9.3 ⭐⭐ 降级的常见策略？**
- 返回默认值/兜底数据/缓存旧数据、关闭非核心功能（大促时关掉推荐/评价）、异步化、直接拒绝部分流量。
- 主动降级（预案开关，人工触发）vs 自动降级（熔断/负载触发）。
- 🧠 类比：降级像"餐厅高峰期简化菜单"——只保证主菜（核心功能）上齐，甜点、饮品（非核心）暂停供应，保证大多数人能吃上饭。
- 延伸：降级前先想清楚"哪些是核心链路，哪些可以牺牲"，这需要业务分级。

**9.4 ⭐⭐⭐ 如何做全链路压测和容量评估？**
- 全链路压测：用影子库/影子表隔离压测数据（不污染真实数据）、流量染色标记压测流量、逐步加压观察 QPS/RT/错误率的拐点。
- 容量评估：单机压测得单实例极限，乘以实例数再打折（预留冗余系数应对突发和长尾）。
- 🧠 类比：全链路压测像"新桥通车前用装满沙袋的卡车模拟满载测承重"——但要用"假车流（影子数据）"别真放市民上去。容量评估像"知道一部电梯限载 13 人，就按 10 人算冗余"。
- 延伸：只压单接口不够，要压真实链路（含缓存、DB、下游）才能发现木桶短板。

**9.5 ⭐⭐ 如何设计一个秒杀系统的高可用防护？**
- 分层拦截：前端按钮置灰 + 验证码错峰、网关限流、Redis 预扣库存（Lua 原子扣减防超卖）、MQ 削峰把下单异步化、库存最终一致回补、缓存挡读。
- 🧠 类比：秒杀像"演唱会抢票"——层层设卡：先限制点击频率（前端）、门口限流（网关）、发放有限的"抢购资格券"（Redis 预扣），拿到券的人再慢慢排队实际下单（MQ 削峰），避免所有人同时冲向收银台（DB）。
- 延伸：核心思想是**层层过滤、快速失败、把瞬时洪峰转成平缓水流**，绝不能让流量直达数据库。

**9.6 ⭐⭐ 超时控制为什么重要？级联超时如何设计？**
- 无超时的调用会一直占用连接/goroutine，下游慢会导致上游资源耗尽并层层传导 = 雪崩。级联：上游超时应 > 下游各段超时之和，用 context deadline 逐级向下传递并递减。
- 🧠 类比：超时像"约会等人的最长忍耐时间"——不设上限就傻等到天荒地老（资源占死）。级联超时像"总行程 2 小时，那每一段中转的等待时间加起来不能超 2 小时"，越往下游给的剩余时间越少。
- 延伸：超时 + 重试 + 熔断 + 限流 + 降级是高可用"五件套"，通常组合使用。

**9.7 ⭐⭐ 什么是雪崩效应？如何整体防护？（补充）**
- 一个服务/资源故障，导致依赖它的上游因等待/重试而资源耗尽，层层扩散最终整个系统瘫痪。
- 防护组合：超时（不傻等）、熔断（快速失败）、限流（控制入口）、降级（保核心）、隔离（线程池/信号量隔离，故障不扩散）。
- 🧠 类比：雪崩像"多米诺骨牌"——一张倒了带倒一片。隔离（舱壁模式 bulkhead）像"轮船分隔舱"，一个舱进水不至于全船沉没。

## 10. Elasticsearch 与检索

**10.1 ⭐⭐ ES 倒排索引原理？**
- 正常思路是"文档→包含哪些词"（正排），倒排反过来："词→出现在哪些文档"。分词器把文本拆成词项（term），建立 词项→文档列表（posting list）的映射；查询时找 term 命中的文档并按相关性打分（BM25）。
- 🧠 类比：正排像"每本书的目录（这本书讲了啥）"；倒排像"书末的'索引'——某个关键词出现在哪几页"。搜"分布式"时直接翻索引找到所有提到它的页，而不是把每本书从头读一遍。
- 延伸：倒排索引是所有搜索引擎的基础；分词质量直接决定搜索效果（中文分词更复杂）。

**10.2 ⭐⭐ 正排（doc values）和倒排的用途区别？**
- 倒排用于"根据词找文档"（全文检索）；doc values 是列式存储的正排，用于排序、聚合、脚本计算（根据文档找字段值）。
- 🧠 类比：倒排回答"哪些文章提到了 Go"（搜索）；doc values 回答"把这些文章按发布时间排序 / 按作者分组统计"（排序聚合）。两个方向的需求要两套结构。

**10.3 ⭐⭐⭐ ES 写入流程与近实时（NRT）原理？**
- 写入先进内存 buffer + 写 translog（防丢）→ 每隔 refresh_interval（默认 1s）refresh 生成一个可搜索的 segment（所以"近实时"，写完约 1s 才能搜到）→ 定期 flush 把 segment 落盘并清 translog → 后台 merge 合并小 segment。
- 🧠 类比：像"草稿→定时誊抄成可查阅的卡片（refresh，1s 一次所以近实时）→ 装订归档（flush）→ 定期把零散卡片合并成大卡片（merge）"。translog 像"随手记的流水账"，防止誊抄前断电丢数据。
- 延伸：调大 refresh_interval 能提升写入吞吐（代价是搜索延迟变大）；批量导入时可临时关闭 refresh。

**10.4 ⭐⭐ 分片（shard）和副本（replica）如何设计？**
- 主分片数**创建后不可改**（要改只能 reindex），决定数据如何拆分和最大扩展能力；副本可动态增减，提升读吞吐和高可用（主挂了副本顶上）。
- 🧠 类比：主分片像"把一个大仓库的货分到 N 个库房"（几个库房一开始就定死）；副本像"每个库房再复制几份备份仓"，既能分担查询又能容灾。
- 延伸：分片不是越多越好——每个分片有固定开销，过多小分片浪费资源；单分片建议控制在几十 GB。

**10.5 ⭐⭐ 深分页问题？ES 如何优化大偏移翻页？**
- `from + size` 深分页时，每个分片都要取 `from+size` 条再汇总排序，内存和 CPU 暴涨（默认限制 1 万）。用 `search_after`（基于上一页最后一条的排序值做游标）翻页，或 `scroll`（快照式批量导出，适合导数据不适合实时翻页）。
- 🧠 类比：和 MySQL 深分页一个道理——别"翻到第 1000 页"，而是"记住上页最后一条，从它往后取"（search_after=游标）。
- 延伸：面向用户的翻页用 search_after，后台全量导出用 scroll 或 PIT（point in time）。

**10.6 ⭐⭐ ES 和 MySQL 如何数据同步？**
- 双写（简单但一致性差）、订阅 binlog（canal）异步同步（推荐，解耦）、Logstash/自研同步服务。关注顺序、幂等、延迟、失败重试。
- 🧠 类比：ES 是 MySQL 数据的"可搜索副本"，靠"监听 MySQL 的改动日志（binlog）自动跟着更新"最省心，避免业务代码里到处双写。
- 延伸：MySQL 做权威数据源（source of truth），ES 只做检索加速，不一致时以 MySQL 为准。

**10.7 ⭐⭐ ES 深度调优点？**
- mapping 精简（不需分词的用 keyword、不需要的关掉 norms/_source 部分字段）、批量 bulk 写入、合理 refresh_interval、冷热数据分层（热节点 SSD、冷节点廉价存储）、定期 force merge 减少 segment。
- 🧠 类比：keyword vs text 像"这字段是拿来精确匹配（身份证号=keyword）还是全文搜索（文章内容=text）"，选错会又占空间又搜不准。

## 11. Linux、网络协议与排障

**11.1 ⭐⭐ TCP 三次握手、四次挥手？为什么是三次/四次？**
- 三次握手（SYN → SYN+ACK → ACK）：确认双方收发能力都正常，防止历史失效的连接请求突然到达而错误建连。四次挥手：TCP 全双工，两个方向要各自关闭，被动方收到 FIN 后可能还有数据要发，所以 ACK 和 FIN 分开发，共四次。
- 🧠 类比：三次握手像打电话——"喂能听到吗？（SYN）""能，你能听到我吗？（SYN+ACK）""能，开始说吧（ACK）"，双方都确认了收发正常。四次挥手像"我说完了（FIN）→ 我知道了但我还没说完（ACK）→ ...我也说完了（FIN）→ 好的挂了（ACK）"。
- 延伸：为什么不是两次握手——防止旧的重复连接请求让服务端错误建连、浪费资源。TIME_WAIT（2MSL）保证最后的 ACK 可靠到达 + 让旧报文消散。

**11.2 ⭐⭐⭐ 大量 CLOSE_WAIT 说明什么问题？**
- CLOSE_WAIT 是"对方已发 FIN 要关，但我方还没调用 close()"的状态。大量堆积 = **应用代码没正确关闭连接**（忘了 Close、连接被阻塞无法关、连接泄漏）。
- 🧠 类比：对方说"我要挂电话了（FIN）"，你"嗯"了一声（ACK）但一直不挂自己这头（不 close），听筒就一直占着（CLOSE_WAIT 堆积），线路（fd）迟早被占满。
- 延伸：排查方向是代码——response body 没 close、DB/HTTP 连接没归还、defer close 被跳过。这是"我方 bug"的强信号。

**11.3 ⭐⭐ TCP 如何保证可靠传输？拥塞控制？**
- 可靠：序列号 + 确认应答（ACK）、超时重传、滑动窗口、流量控制（接收方通告窗口大小防被压垮）。拥塞控制：慢启动（指数增长探测）、拥塞避免（线性增长）、快重传、快恢复。
- 🧠 类比：拥塞控制像"新手上高速"——先慢慢加速试探路况（慢启动），感觉顺就稳步加速（拥塞避免），一旦发现前方拥堵（丢包）马上减速（快恢复），避免堵死整条路。
- 延伸：流量控制是"别把接收方缓冲区撑爆"（点对点），拥塞控制是"别把网络整体压垮"（全局），两者不同。

**11.4 ⭐⭐ HTTP/1.1、HTTP/2、HTTP/3 的区别？**
- 1.1：文本协议、长连接（keep-alive）但有队头阻塞（一个响应慢堵住后面）。2：二进制分帧、多路复用（一个连接并发多请求）、头部压缩（HPACK）、服务端推送。3：基于 QUIC（UDP），解决 TCP 层的队头阻塞、连接迁移快、握手更快。
- 🧠 类比：1.1 像"单车道，前车慢后车全堵"；2 像"多车道但共用一条路基，路基（TCP）出坑所有车道都受影响（TCP 队头阻塞）"；3 换成"每条车道独立路基（UDP+QUIC）"，一条坑了不影响其他。
- 延伸：HTTP/2 解决了应用层队头阻塞，但 TCP 层丢包仍会阻塞所有流，HTTP/3 用 QUIC 彻底解决。

**11.5 ⭐⭐ 常用排障命令？**
- `top/htop`（CPU/内存/进程）、`vmstat`（系统整体）、`iostat`（磁盘 IO）、`free`（内存）、`ss/netstat`（连接状态）、`tcpdump`（抓包）、`lsof`（打开的文件/连接）、`strace`（系统调用跟踪）、`perf`（性能剖析）、`dmesg`（内核日志，看 OOM kill）、`pidstat`。
- 🧠 类比：这些命令像"医生的听诊器、血压计、X 光机"——top 量整体状态，iostat 查磁盘"血压"，tcpdump 是"网络 X 光"，strace 是"看它到底在跟系统要什么"。
- 延伸：记忆分类——看 CPU（top/pidstat）、看内存（free/vmstat）、看 IO（iostat/iotop）、看网络（ss/tcpdump）、看行为（strace/lsof）。

**11.6 ⭐⭐⭐ 一台机器 CPU 打满如何逐步定位？**
- `top` 找占用高的进程 → `top -Hp <pid>` 找占用高的线程 → 结合 pprof（Go）/ perf / 火焰图定位到函数。看 CPU 构成：`us`（用户态）高 → 看应用代码；`sy`（内核态）高 → 看系统调用/上下文切换；`wa`（IO 等待）高 → 看磁盘/网络 IO。
- 🧠 类比：像医生查"发烧原因"——先定位哪个器官（进程）发热，再细查哪块组织（线程/函数），再判断是自身炎症（us 用户态代码）还是外部感染（sy 系统调用/wa IO）。
- 延伸：Go 服务直接上 pprof cpu profile + 火焰图，比 perf 更精准到函数。

**11.7 ⭐⭐ Load 高但 CPU 不高，可能原因？**
- Load 统计的是"可运行 + 不可中断（D 状态，通常等 IO）"的进程数。CPU 不高但 load 高，多半是大量进程卡在 IO 等待（磁盘慢、网络慢、NFS 卡）。用 `iostat`、`iotop`、看 D 状态进程排查。
- 🧠 类比：load 高 CPU 不高像"餐厅排长队但厨师没在忙"——不是厨房（CPU）慢，是大家都在等某个上游供货（IO），队伍（load）越排越长。
- 延伸：Linux 的 load 包含 D 状态（不可中断睡眠）是它区别于其他系统的关键，所以 load 高≠CPU 忙。

**11.8 ⭐⭐ 文件描述符耗尽（too many open files）如何处理？**
- 调高 `ulimit -n` / systemd LimitNOFILE；用 `lsof -p <pid>` 查是什么占满（连接泄漏？文件没关？）；根治靠复用连接池 + 确保资源关闭。
- 🧠 类比：fd 像"公司的门禁卡总数"——发完了新人进不来（无法新建连接/打开文件）。要么多印卡（调 ulimit），要么查是谁拿了卡不还（连接泄漏）。
- 延伸：每个 socket/文件都占 fd，连接泄漏（见 CLOSE_WAIT）是常见根因。

**11.9 ⭐⭐ 零拷贝、mmap、sendfile 的作用？**
- 传统读文件发网络要经历"磁盘→内核缓冲→用户缓冲→socket 缓冲→网卡"多次拷贝和上下文切换。零拷贝（sendfile/mmap）省去用户态那趟，数据在内核内直接流转。
- 🧠 类比：普通方式像"仓库发货先搬到你办公室，你再搬去快递点"（多搬一趟）；零拷贝像"仓库直接发给快递点"，省掉中间搬运。
- 延伸：Kafka、Nginx 静态文件、文件下载服务都靠零拷贝提升吞吐。

**11.10 ⭐⭐ epoll 相比 select/poll 的优势？**
- select/poll 每次调用都要把所有 fd 传入内核并**遍历全部**判断谁就绪（O(n)）；epoll 用事件回调 + 就绪列表，只返回真正就绪的 fd（O(1)），支持海量连接。
- 🧠 类比：select 像"点名时把全班名字从头念一遍看谁在"（人越多越慢）；epoll 像"谁到了自己在签到表登记，老师只看签到表"（不管班多大都高效）。
- 延伸：epoll 是 Go netpoller、Nginx、Redis 高并发网络的底层基础（Linux 上）。

## 12. Git 工作流与 CI/CD 工程化

**12.1 ⭐ 常见分支模型？**
- Git Flow（feature/develop/release/hotfix/master，规范但重）、GitHub Flow（主干 + feature 分支 + PR，轻量）、Trunk-Based（几乎都在主干开发 + 特性开关，适合高频发布/CI 成熟团队）。
- 🧠 类比：Git Flow 像"大公司严格的多级审批流程"（稳但慢）；Trunk-Based 像"小团队直接在主线上快速迭代"（快但要求测试/开关到位）。
- 延伸：分支越长命越难合（合并冲突/集成地狱），现代趋势是短命分支 + 频繁集成。

**12.2 ⭐⭐ rebase 和 merge 的区别？何时用哪个？**
- merge：保留完整历史，产生一个 merge commit，历史呈网状；rebase：把你的提交"搬到"目标分支最新提交之后，历史线性整洁，但**改写了提交**。
- 原则：**公共/共享分支禁止 rebase**（改写他人依赖的历史会天下大乱）；本地私有分支整理提交、同步主干用 rebase。
- 🧠 类比：merge 像"如实记录两条路怎么汇合的（有个交汇点）"；rebase 像"假装我一直是接着最新进度做的，把历史捋直"。别对已经发给别人的历史 rebase（等于偷偷改了大家共同的账本）。
- 延伸：`git pull --rebase` 保持提交线性；出问题用 `git reflog` 找回。

**12.3 ⭐⭐ 如何做 code review？关注点有哪些？**
- 关注：正确性、边界条件、并发安全、错误处理是否完整、可读性/命名、测试覆盖、性能隐患、安全（注入/越权/密钥硬编码）、是否符合团队规范。实践：小 PR（易审）、及时反馈、对事不对人、自动化（lint/测试）先行把关。
- 🧠 类比：code review 像"论文同行评审"——不只挑错，还帮作者提升；PR 太大像"一次交 200 页论文"没人愿意认真看。
- 延伸：作为专家/带人角色，review 更要引导而非直接改，帮初级理解"为什么"。

**12.4 ⭐⭐ CI/CD 流水线一般包含哪些阶段？**
- 拉代码 → 静态检查（lint / go vet / 安全扫描）+ 单元测试（含 -race）→ 构建产物/镜像 → 推镜像仓库 → 部署（灰度/蓝绿/金丝雀）→ 冒烟/回归测试 → 观测告警。
- 🧠 类比：CI/CD 像"食品自动化流水线 + 质检关卡"——每道工序（提交）都过安检（测试/扫描），不合格的当场拦下，合格才进下一环节直到出厂（上线）。
- 延伸：CI（持续集成）重在"频繁合并 + 自动验证"，CD（持续交付/部署）重在"自动化、可重复、可回滚的发布"。

**12.5 ⭐⭐ 灰度发布、蓝绿部署、金丝雀发布的区别？**
- 蓝绿：准备两套完整环境（蓝=旧、绿=新），流量一次性切换，回滚快但要双份资源。金丝雀：先放极小流量到新版本试探，逐步放量。灰度：按用户/地域/比例分批放量。
- 🧠 类比：金丝雀发布名字来自"矿工带金丝雀下矿探毒气"——先让一小批用户"试毒"，没问题再全量。蓝绿像"两个一模一样的舞台，灯光（流量）一键切换，出问题立刻切回"。
- 延伸：金丝雀/灰度能在小范围暴露问题，是降低发布风险的主流手段；配合监控自动判断是否继续放量。

**12.6 ⭐⭐ 如何保证发布可回滚、可观测？**
- 版本化不可变镜像、数据库变更向后兼容（先加兼容字段→发布→再清理，避免 schema 与代码不匹配）、特性开关控制、完善的监控/告警/日志 + 快速回滚预案（一键回退上一个稳定版本）。
- 🧠 类比：可回滚像"游戏存档"——每次发布都存个能秒回的档，出事就读档。数据库兼容变更像"先修好新旧都能走的路，再拆旧路"，不能一步到位把旧代码的路挖断。
- 延伸：数据库"先兼容后清理"（扩展-收缩模式 expand-contract）是发布安全的关键，直接改字段/删列会让回滚时旧代码崩溃。

**12.7 ⭐⭐ Go 项目工程化：依赖管理、lint、测试如何组织？**
- go modules 锁版本（go.mod/go.sum）；golangci-lint 统一静态检查；表驱动测试 + `-race` + 覆盖率；接口 + mock 解耦依赖；集成测试用 testcontainers 起真实中间件。
- 🧠 类比：go.sum 像"依赖的防伪校验码"，保证每次拉到的依赖和当初一模一样（可复现构建）。表驱动测试像"一张测试用例表格，一行一个 case"，加用例只需加一行。

## 13. 系统设计（高并发场景实战）

> 通用方法论（万能框架）：**需求澄清 → 容量估算（QPS/存储/带宽）→ 核心接口与数据模型 → 存储选型 → 缓存与一致性 → 高可用与水平扩展 → 瓶颈分析与权衡取舍**。
> 🧠 记忆：先问清楚要什么（需求）→ 算多大（容量）→ 怎么存（模型/存储）→ 怎么快（缓存）→ 怎么稳（高可用）→ 哪里会挂（瓶颈）。

**13.1 ⭐⭐⭐ 设计一个短链系统。**
- 发号（雪花/号段）→ base62 编码成短码；读多写少 → Redis 缓存热点 + 布隆过滤器挡无效码；访问用 302 跳转；防刷限流；点击统计异步化（MQ）。
- 🧠 类比：短链像"给长网址发一个简短的会员号"，核心是"发号 + 编码 + 高速查表跳转"。

**13.2 ⭐⭐⭐ 设计一个高并发计数/点赞系统。**
- Redis 原子自增扛写 + 定时/批量异步落库；热点计数打散（分桶累加后合并）；用户维度去重防重复点赞；最终一致。
- 🧠 类比：像"演唱会实时人数计数器"——不可能每进一人就写一次数据库，先在内存快速累加，定期同步到账本（DB）。

**13.3 ⭐⭐⭐ 设计一个 Feed / 时间线系统。**
- 推模式（写扩散：发帖时写入所有粉丝的收件箱，读快写重，适合普通用户）vs 拉模式（读扩散：读时实时聚合关注者的帖子，写轻读重，适合大 V）vs 推拉结合（普通用户推、大 V 拉）。
- 🧠 类比：推模式像"群发短信到每个粉丝信箱"（发一次累死，但粉丝看信箱很快）；拉模式像"粉丝自己去每个关注对象主页翻"（大 V 有千万粉就不能推）。微博这种就用推拉结合。
- 延伸：这是"写扩散 vs 读扩散"的经典权衡，答题要点出"根据用户粉丝量分层处理"。

**13.4 ⭐⭐⭐ 设计一个分布式延迟队列 / 定时任务。**
- 方案：Redis ZSet（score = 执行时间戳，轮询到期的）、时间轮（高效管理海量定时器）、RocketMQ 延迟级别、DB 扫表（简单但不精准）。关注幂等、避免重复触发、水平扩展下的任务分片。
- 🧠 类比：ZSet 延迟队列像"按'该叫醒的时间'排序的闹钟列表"，轮询看有没有到点的；时间轮像"钟表的齿轮"，指针转到哪格就触发那格的任务，管理海量定时器很省。

**13.5 ⭐⭐⭐ 设计一个 IM / 消息推送系统。**
- WebSocket 长连接网关（维护在线连接）+ 路由层（用户→连接在哪台机器）+ 在线状态管理 + 消息可靠投递（ack + 重试 + 去重 + 离线存储）+ 群消息扩散。
- 🧠 类比：IM 像"邮局 + 实时对讲机"——在线的直接对讲（长连接推送），不在线的存到信箱（离线消息），上线后补发。核心难点是"找到对方现在连在哪台服务器"（路由）和"消息不丢不重"。

**13.6 ⭐⭐⭐ 设计一个订单/支付系统的核心链路。**
- 幂等下单（防重复提交）、库存扣减防超卖（Redis Lua / DB 乐观锁）、支付回调对账、超时未支付关单（延迟队列）、状态机管理订单状态流转、分布式事务/最终一致保证钱货一致。
- 🧠 类比：订单系统像"严谨的收银流程"——每一步状态明确（下单→待支付→已支付→发货，状态机），钱和货必须对得上账（对账），同一笔不能重复扣款（幂等）。
- 延伸：钱相关系统强调"幂等 + 对账 + 状态机 + 最终一致"，可靠性优先于性能。

**13.7 ⭐⭐⭐ 亿级流量下如何做系统重构 / 架构演进？**
- 演进路径：单体 → 垂直拆分 → 服务化/微服务 → 中台化。方法：识别瓶颈分阶段迁移、新旧并行双跑对比结果、灰度切流、数据迁移用双写 + 校验 + 回滚预案，保证不停机平滑过渡（绞杀者模式 strangler）。
- 🧠 类比：重构大系统像"给正在飞的飞机换引擎"——不能停机，得一个引擎一个引擎地换（分阶段），换的时候新旧引擎并行验证（双跑），确认新的没问题再拆旧的（绞杀者）。
- 延伸：核心原则——**小步快跑、可灰度、可回滚、数据可校验**，绝不"推倒重来一把梭"。

## 14. 线上故障排查与复盘

**14.1 ⭐⭐⭐ 一次线上接口大面积超时，你的排查思路？**
- 分层定位：监控大盘看现象（RT/错误率/QPS 突变）→ 判断是自身还是下游（DB/缓存/RPC/MQ 哪层慢，看链路追踪）→ 查资源（CPU/内存/GC/连接池/FD/网络）→ 查变更（最近发布？配置改动？流量突增？）。
- **原则：先止血（回滚/降级/限流/扩容）恢复业务，再从容定位根因**——别在故障时纠结根因让损失扩大。
- 🧠 类比：像医院急诊——先止血保命（恢复服务），稳定后再做详细检查找病因（根因分析）。慌乱中最忌讳"边流血边研究为什么流血"。
- 延伸：排查口诀记忆——**看现象、分上下（自身/下游）、查资源、追变更**；80% 的线上问题和"最近的变更"有关，先看变更效率最高。

**14.2 ⭐⭐⭐ 服务频繁 OOM 被 kill，如何排查？**
- pprof heap 看内存分布；核对容器 memory limit 与 GOMEMLIMIT/GOGC 配比；查大对象、goroutine 泄漏、无上限的缓存/map、大量并发导致的瞬时分配；`dmesg` 看是否被 OOM killer 杀。
- 🧠 类比：OOM 像"房间被塞爆"——先看是"长期堆积（泄漏）"还是"瞬间涌入太多东西（并发高峰分配）"，再决定是修漏还是限流/扩容/设内存红线（GOMEMLIMIT）。
- 延伸：容器里 Go 不设 GOMEMLIMIT 很容易 OOM——因为 Go 默认按堆的 2 倍才 GC，可能超过容器 limit 被杀。

**14.3 ⭐⭐ goroutine 数量暴涨如何定位？**
- pprof goroutine profile 看堆栈聚集在哪一行——通常是某个下游阻塞导致请求积压、channel 没有出口、缺少超时控制。
- 🧠 类比：goroutine 暴涨像"客服电话全部占线还在不停接新客户"——多半是某个环节（下游）卡住不放，新请求进来就堆着出不去。
- 延伸：几乎总能在 pprof 里看到几万个 goroutine 停在同一个栈帧（如某个 channel 接收或 RPC 调用），那就是元凶。

**14.4 ⭐⭐ 数据库连接池打满怎么办？**
- 排查慢查询/长事务/未释放连接（连接泄漏）；合理配置 maxOpenConns / maxIdleConns / connMaxLifetime；给 SQL 加超时；读写分离/加缓存分流。
- 🧠 类比：连接池像"银行有限的柜台"——柜台全占着（打满），要么是有人办业务超慢（慢查询/长事务），要么是有人办完不走（连接没归还）。
- 延伸：连接池不是越大越好，过大反而压垮 DB；要结合 DB 的最大连接数规划。

**14.5 ⭐⭐⭐ 复盘（Postmortem）应该包含什么？**
- 时间线（从发生到恢复）、影响范围（多少用户/多少钱）、根因（5 Why 深挖到根）、止血与恢复过程、改进项（可落地、有 owner、有截止时间），文化上**对事不对人（blameless）**。
- 🧠 类比：复盘像"空难调查"——目的不是追责怪飞行员，而是找出系统性隐患防止再次发生（blameless）。用 5 Why 层层追问："为什么超时？→ 因为 DB 慢 → 为什么慢？→ 因为没索引 → 为什么没发现？→ 因为缺少慢查询监控…"直到根因。
- 延伸：好的复盘产出的是"系统性的、有主人的、有时限的改进项"，而不是"下次注意"这种空话。

## 15. 软性能力与远程协作（行为面）

> 通用答题框架：**STAR**（Situation 情境 → Task 任务 → Action 行动 → Result 结果，结果尽量量化）。

**15.1 讲一个你主导的、最有挑战的核心项目，你的角色和难点攻坚？**
- 关键点：清晰交代背景和你的角色（主导/核心）、难点在哪、你做了哪些关键决策和权衡、量化结果（QPS 提升 X 倍、P99 从 A 降到 B、成本降 X%）。
- 🧠 记忆：讲项目要突出"**你的**决策和取舍"，而不是流水账；面试官想看的是你的判断力，不是项目本身多牛。

**15.2 你做过最复杂的性能优化案例？收益如何量化？**
- 关键点：怎么定位瓶颈（pprof/监控数据说话）、对比了哪几种方案、为什么选这个、优化前后数据对比、有没有副作用/权衡。
- 🧠 记忆：性能优化题的灵魂是"**先测量再优化**"——能说出"我通过 xx 工具发现瓶颈在 yy"比"我觉得这里慢"专业得多。

**15.3 一次线上重大事故，你是如何处理和复盘的？**
- 关键点：冷静先止血、主动承担、系统性改进（不是打补丁）、blameless 复盘、落地了哪些长效改进。
- 🧠 记忆：面试官看的是"你面对压力的处理方式和责任心"，别甩锅、别只讲技术细节忽略协作。

**15.4 技术选型有分歧时你如何推进决策？**
- 关键点：用成本/收益/风险分析、数据和 POC 说话、倾听不同意见、达成共识、把决策和理由记录下来（ADR 架构决策记录）。
- 🧠 记忆：专家不是"我说了算"，而是"用事实和数据让大家心服口服"，并对决策负责、留档可追溯。

**15.5 如何带初级开发做技术传帮带？**
- 关键点：通过 code review 引导思考（多问"为什么"少直接给答案）、沉淀文档、结对编程、合理拆解任务给成长空间、容忍可控的试错。
- 🧠 记忆：带人不是"帮他把活干了"，而是"让他学会自己干"——授人以渔。

**15.6 全职远程你如何保证自律和高效协作？**
- 关键点：结果导向（对齐目标和交付）、异步沟通（清晰的文字/文档/记录减少同步依赖）、主动同步进度（不让人追着问）、约定响应时效和重叠时间、可复盘可追溯（重要决策落文字）。
- 🧠 记忆：远程协作的核心是"**主动 + 透明 + 文档化**"——让协作者随时能知道你在做什么、进展如何，不靠"看到你在工位"来建立信任。

**15.7 你如何持续跟进前沿技术并落地到业务？**
- 关键点：稳定的信息源、判断标准（是否解决真实业务痛点，而非追新）、小范围 POC 验证、评估引入成本/风险/团队学习曲线，再决定是否推广。
- 🧠 记忆：专家对新技术是"**冷静评估、小步验证**"而非"看到就上"——技术服务于业务，不是炫技。

## 附：高频「一句话」速记卡

- **Go 并发哲学**：不要通过共享内存来通信，要通过通信来共享内存。
- **GMP**：G 任务、M 工人、P 工位；work-stealing 偷任务；网络 IO 走 netpoller 不占 M。
- **GC**：三色标记 + 混合写屏障 → 亚毫秒 STW；GOGC 控频率，容器设 GOMEMLIMIT。
- **逃逸**：被外部还要用（返回指针/闭包/interface）就逃到堆；减少逃逸=减轻 GC。
- **缓存三问**：穿透查无（布隆/空值）、击穿单点热 key（互斥/逻辑过期）、雪崩批量过期（随机 TTL/高可用）。
- **缓存一致性**：Cache Aside + 先更库后删缓存 + 延迟双删 + binlog 订阅。
- **消息可靠**：生产 acks=all、Broker 多副本、消费先处理后手动提交 + 幂等去重（至少一次+幂等=恰好一次）。
- **Kafka 顺序**：单分区有序，同 key 同分区。
- **MySQL 优化**：EXPLAIN 看 type/key/rows/Extra；覆盖索引免回表；最左前缀；范围列后失效。
- **三大日志**：redo（崩溃恢复/物理）、undo（回滚/MVCC）、binlog（复制/逻辑）；两阶段提交保一致。
- **深分页**：游标（记住上次 id）+ 延迟关联（先查主键）。
- **分布式事务**：能最终一致就别强一致；2PC 强一致阻塞、TCC 业务侵入、Saga 补偿、消息表异步。
- **限流四算法**：固定窗口、滑动窗口、漏桶（平滑）、令牌桶（允许突发）。
- **高可用五件套**：超时、重试（退避+幂等）、熔断、限流、降级 + 隔离。
- **CAP**：分区时才在 C 和 A 间取舍；BASE=基本可用+软状态+最终一致。
- **排障口诀**：先止血后定位；看现象→分自身/下游→查资源→追变更（80% 和最近变更有关）。
- **CLOSE_WAIT 多**=我方没 close（代码 bug）；**TIME_WAIT 多**=主动关闭方正常现象。
- **Load 高 CPU 不高**=大量进程等 IO（D 状态）。
- **发布安全**：可灰度、可回滚、DB 变更先兼容后清理（expand-contract）。
- **行为面**：STAR 框架 + 量化结果 + 突出你的决策和权衡。
